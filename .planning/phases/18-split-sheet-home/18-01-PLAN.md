<objective>
Give the split sheet a home it can be returned to. Today a saved draft is permanently unreachable: there is no list, no detail page, and no edit mode, so PATCH /api/split-sheets/[id] has never had a caller (18-CONTEXT findings 1 and 2). This plan builds the living-draft surface — list, detail/edit page, builder edit mode with the collaborator picker available on an existing draft (P18-07), add-and-redistribute, a read-only share that is explicitly not a formal ask (P18-08), and change summaries when consensus resets (P18-09) — all of it respecting and explaining the freeze boundary already shipped in lib/split-sheets/lifecycle.ts rather than working around it (P18-06).

Purpose: every other plan in this phase assumes a sheet can be looked at. The Locker's attention feed (18-02) links to sheets; attachment (18-03) acts on them. Without a reachable sheet the rest of the phase points at nothing.
Output: two pure modules with tests, a widened list query, a share route, three pages, and SplitSheetBuilder taught to edit.
</objective>

  <behavior>
    redistribute(splits, mode):
    - mode 'even' — every party receives evenSplit(n); the existing helper in lib/split-sheets/approval.ts is the single source for that number.
    - mode 'proportional' — each existing party keeps its relative weight, scaled so the whole set including the new party totals exactly 100.000. A set of 50/30/20 gaining a fourth party scales the first three down while preserving their 5:3:2 ratio.
    - Remainder handling: rounding is to three decimal places and any residue is applied to the largest share, so validateApprovalTotal() returns true for every output of this function. This is asserted for party counts 1 through 12 in both modes.
    - Removing a party redistributes the freed percentage by the same two modes.
    - A zero-total or empty input returns an even distribution rather than dividing by zero.

    summarizePartyChanges(before, after):
    - Returns an ordered list of structured change records, each carrying a kind ('added' | 'removed' | 'moved'), the party name, and for 'moved' the from and to percentages.
    - A party present in both sets with an unchanged percentage produces no record.
    - Matching between the two sets is by party id when present, falling back to normalized name — the same normalization discipline lib/split-sheets/reconciliation.ts already uses for name matching.
    - Every record renders through a formatter in this module that produces system-worded English only. The function accepts no caller-supplied message string and has no parameter through which one could be passed (P18-13).

    resolvePartyPhase() gains a 'preview' branch:
    - When the sheet status is draft, a valid token resolves to 'preview' regardless of the party's own approval_status.
    - Every existing branch keeps its current result for every non-draft status; the existing phase.test.ts cases must continue to pass unmodified.
  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| party session → sheet list read | A cross-user read; a party must reach shared sheets without reaching another user's unsent drafts. |
| initiator session → share mint | Mints durable public tokens that expose party legal names, PRO and IPI to whoever holds the URL. |
| public token → preview render | An unauthenticated surface reading a sheet the visitor does not own. |
| initiator session → party-set PATCH | Edits that destroy prior approvals and, past the boundary, could corrupt a signed legal record. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-01 | Information Disclosure | GET /api/split-sheets returning another user's working draft | high | mitigate | Draft-status sheets are returned to the initiator only; the rule is applied in the server merge, never in the client, and asserted in the route's own test. |
| T-18-02 | Information Disclosure | share tokens leaking co-party legal name, PRO and IPI | medium | accept | Co-party detail disclosure is by design on a shared agreement (design section 10b), and the token carries the same 30-day expiry and 256-bit entropy as the approval token it reuses. Accepted with the existing expiry as the bound. |
| T-18-03 | Elevation of Privilege | share route used to advance the lifecycle or to send on a sheet the caller does not own | high | mitigate | Initiator-only authorization, a 409 on any status other than draft or countered, and no status write anywhere in the handler. |
| T-18-04 | Tampering | editing past the freeze boundary destroying approval tokens or e-sign signer linkage | critical | mitigate | The UI consumes assertEditable() from lib/split-sheets/lifecycle.ts and renders its refusal rather than deriving its own; the server gate in the PATCH route remains the enforcement point and is not bypassed by any new client path. |
| T-18-05 | Repudiation | a consensus reset that leaves parties unable to tell what they are re-approving | medium | mitigate | summarizePartyChanges() produces a named, from/to change record set shown to the initiator on reset and carried by the reset notification. |
| T-18-06 | Information Disclosure | free text riding a share or reset notification across a block | high | mitigate | Neither the share route nor the change-summary module accepts any caller-supplied string; the absence is structural, documented at both files citing design section 10c-ii (P18-13). |
| T-18-SC | Tampering | npm/pip/cargo installs | low | accept | No package installs in this plan. |
</threat_model>
