<objective>
Make a split sheet song-specific and make attachment a relationship rather than a move. A split sheet governs one composition, and a five-track EP nearly always has five different split configurations, but split_sheets has no track_id and 17-05's attach route can only target a project (design gaps 1 and 2). This plan adds the column and the split_sheet_attachments join table with a backfill (design section 2), extends 17-05's route to accept a track and relaxes its executed-only gate per P18-04, builds attach from both directions with fuzzy title matching (design section 3), adds detach, and handles the section 7 edge cases explicitly. It also ships the source provenance field so provenance exists from day one (P18-05).

Purpose: without song-level attachment, coverage-based readiness has nothing to measure and the Locker-first path — the path that matches when split sheets actually get signed, at 2am before a release exists — stays second-class.
Output: migration 064 with tests, a pure attachment module, attach v2 plus detach, and attach UI from the Locker side and the Vault side.

Wave placement: this plan depends on nothing and shares no file with 18-01 or 18-02, so it could run in wave 1. It is placed in wave 2 deliberately, to serialize migration authoring — 064 must be claimed and settled before 18-04 claims 065 — and because both attach surfaces link into routes 18-01 and 18-02 create in the waves at or before this one. The empty depends_on is therefore correct and the wave number is a scheduling choice, not a missing dependency.

Two design open questions are resolved here, both as the design leaned. May a sheet attach before execution: YES — 17-05's executed-only gate is relaxed, because a mid-approval sheet showing on the project as in progress is more useful than invisible, and P18-04 makes attachment lifecycle-orthogonal by construction. Does attaching notify the other parties: NO for v1 — attaching is an organizational act, not a legal one, and a notification here would also be the first cross-party emission this plan could get wrong under P18-13.
</objective>

  <action>
    Author supabase/migrations/064_split_sheet_song_attachment.sql following the conventions migration 062 establishes: a header block naming the migration, its decision references, and the standing instruction that an executor agent must never run the database push — the live push is the human-gated checkpoint at the end of this plan.

    The uniqueness requirement has a trap in it worth stating plainly. A plain unique constraint over the sheet/project/track triple does not prevent a duplicate when track_id is null, because null values compare as distinct in a Postgres unique index. Since the project-level attachment — the covers-the-whole-release case — is exactly the case where track_id is null, a naive constraint leaves the one case it most needs to guard wide open. Use a pair of partial unique indexes: one over all three columns where track_id is not null, one over sheet and project where track_id is null.

    Choose the cascade behavior deliberately and comment why at each foreign key, because two of the four are the opposite of each other on purpose. The attachment row cascades away when its project is deleted, so the sheet returns to unattached in the Locker rather than pointing at nothing. The track reference nulls rather than cascading everywhere it appears, so a deleted track leaves the attachment alive at project level and leaves the legal record of who wrote the song completely untouched. Deleting a track must never delete the record of who wrote it, and that sentence is the whole rationale for both choices.

    Enable row level security on the new table and follow the server-owned write doctrine migrations 040, 056, 058 and 062 established: revoke insert, update and delete from the authenticated and anonymous roles so attachment happens only through the service-role route in Task 2 after its own authorization checks. Model the select policies on migration 018's initiator-sees-all plus party-sees-own pair, and add the project-owner read as a third path so a project owner can see what is attached to their release.

    Write the backfill as a single insert-select from split_sheets where vault_project_id is not null, made idempotent so a re-run is a no-op. Leave split_sheets.vault_project_id and track_id in place as the origin fields — where the sheet was born — rather than dropping them; design section 2b keeps them and the attachments are the relationship set.

    Write __tests__/migration-064.test.ts as a string-assertion test in the exact style of __tests__/migration-063.test.ts: read the file and assert each structural element is present, including both partial indexes, both cascade behaviors, the revoke, and the idempotent backfill.
  </action>

<task type="auto" tdd="true">
  <name>Task 2: The attachment module, attach v2, and detach</name>
  <files>lib/split-sheets/attachment.ts, lib/split-sheets/attachment.test.ts, app/api/split-sheets/[id]/attach/route.ts, app/api/split-sheets/[id]/detach/route.ts</files>
  <behavior>
    lib/split-sheets/attachment.ts:
    - suggestTrackMatches(songName, tracks) returns the project's tracks ordered by title similarity to the sheet's song name, each with a score, and marks a leading candidate only when it clears a confidence threshold. Comparison is on normalized titles — case, punctuation and surrounding whitespace folded, and common decorations such as a parenthesized remix or feature suffix discounted rather than treated as a mismatch.
    - A rename produces a weak or zero match rather than a wrong confident one; the function must never mark a leading candidate it is not confident in, because a wrong confident suggestion on a legal document is worse than no suggestion.
    - detectTrackConflicts(attachments) returns every track id claimed by more than one sheet, with the claiming sheet ids. It reports; it never resolves.
    - describeSignedTitle(sheetSongName, currentTrackTitle) returns a structured record describing whether the titles diverge and what each is, for the renamed-after-signing display. It returns data, not a rendered sentence, and it never implies the document should be reissued.

    POST /api/split-sheets/[id]/attach accepts a project id and an optional track id:
    - 401 without a session; 403 unless the caller is a party on the sheet AND owns the destination project. This double check is unchanged from 17-05 and is not loosened.
    - The executed-only status gate is REMOVED. A sheet at any status may attach (P18-04).
    - When a track id is supplied, it is verified to belong to the destination project before any write; a track from another project is a 403 under the same generic rejection shape the route already uses.
    - Writes an attachment row, sets the sheet's origin fields when they were previously null, and updates the caller's own split-sheet vault_documents row's project and track so the readiness trigger fires. It never duplicates a document row and never writes another party's row.
    - Attaching a sheet already attached to a different project succeeds and produces a second attachment row; this is the single-and-album case and is the reason the join table exists.
    - Re-attaching an identical sheet/project/track triple is a no-op success, not a duplicate row and not an error.
    - The response body is a status result. It emits no notification and accepts no note, reason or message field.

    POST /api/split-sheets/[id]/detach:
    - Same double authorization check.
    - Removes the attachment row and nulls the caller's own document row's project and track.
    - Issues no delete against split_sheets, split_sheet_parties, vault_documents, or storage. Readiness recalculating downward afterwards is the correct outcome.
  </behavior>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| party session → attach write | A cross-user action associating a shared legal document with a release; requires party-and-owner authorization. |
| caller-supplied track id → project scope | An identifier from the client that must be proven to belong to the destination project before any write. |
| detach → shared legal record | A destructive-sounding action on a document co-owned by other parties. |
| migration backfill → existing associations | A one-shot data movement over live legal associations. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-13 | Elevation of Privilege | attaching a sheet to a project the caller does not own | high | mitigate | 17-05's party-and-owner double check is preserved verbatim and re-asserted by the route's authorization-matrix test; relaxing the status gate does not touch the ownership gate. |
| T-18-14 | Tampering | a track id from another project written onto an attachment | high | mitigate | The track is verified to belong to the destination project with the session client before any service-client write; failure returns the route's existing generic rejection. |
| T-18-15 | Denial of Service | detach or a track deletion destroying a signed agreement | critical | mitigate | Detach deletes only the attachment row; track references are ON DELETE SET NULL so a deleted track leaves the attachment and the sheet intact; the non-destruction property is asserted by test rather than by inspection. |
