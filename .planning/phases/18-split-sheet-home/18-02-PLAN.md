<objective>
Turn the Contract Locker from a filing cabinet into a workspace (design section 10). The substance of this plan is a query change, not a layout change: the Locker today renders only vault_documents — signed artifacts — so every draft and every sheet awaiting signature is invisible to it, and those are exactly the things that need attention (18-CONTEXT finding 3). This plan teaches the page to read in-flight split_sheets as well, derives the attention-first sections from pure structured queries (P18-10), scopes every row to the viewing party's own context with drafts held back until sent and removal implemented as a per-party hide (P18-11), and makes the block exception deliberate by documenting it at the query site (P18-12).

Purpose: the Locker is the front door for the entire split-sheet product. A front door that can only see finished paperwork cannot tell an artist that three people still owe them a signature.
Output: a pure attention-derivation module with tests, the widened page query with the documented block exception, a per-party hide route, and the rebuilt Locker landing.
</objective>

  <name>Task 1: buildAttentionSections — the pure, model-free derivation</name>
  <files>lib/contracts/locker-attention.ts, lib/contracts/locker-attention.test.ts</files>
  <behavior>
    buildAttentionSections({ viewerUserId, sheets, documents, projects, hiddenDocumentIds }):
    - Returns four ordered attention sections plus a settled archive set. The order is fixed by P18-10 and is part of the contract: awaiting signature, drafts in progress, unattached executed, songs with no sheet.
    - Awaiting signature: sheets in pending_approval, countered, approved or esign_pending. Each carries per-party progress — how many parties have acted out of how many, and each party's own state.
    - Drafts in progress: sheets in draft status where viewerUserId is the initiator. A draft whose initiator is someone else is excluded from every section and from the archive; it does not exist for this viewer.
    - Unattached executed: executed sheets with no release association. Phrased as a nudge, never as a warning — an unreleased song's sheet is a valid permanent state (design section 7).
    - Songs with no sheet: the viewer's project tracks with no split sheet associated to them.
    - Settled archive: executed sheets and signed documents that are not in any attention section.
    - Viewer context: every row carries the viewer's OWN share percentage and their OWN signing state, resolved from the party row whose user_id matches viewerUserId. When the viewer is the initiator but not a named party, the share is null rather than another party's number.
    - Any document id in hiddenDocumentIds is omitted from every section for this viewer only; the function has no ability to express hiding for anyone else.
    - The module performs no I/O and calls no model. Its inputs are plain row arrays.
  </behavior>
  <action>
    Create lib/contracts/locker-attention.ts implementing the behavior block as a pure module in the style of lib/vault/readiness-tiers.ts — named exports, exported types, a header comment explaining why the module exists. The reason it exists is worth writing down: the highest-value version of this landing is deterministic and instant because Funūn generated the underlying data, and inferring what is already known would be slower and less correct (design section 10a).

    Consume the status vocabulary from lib/split-sheets/lifecycle.ts rather than re-declaring string literals, so a future lifecycle change cannot leave this module silently mis-bucketing a status it no longer recognizes. An unrecognized status must fall into the archive rather than throw, matching the degradation posture deriveSheetTier() already takes.

    The viewer-context resolution is the P18-11 heart of this module and deserves its own exported function and its own test cases: one agreement viewed by two different parties must produce two different share figures from the same input rows, and neither viewer may ever see the other's figure presented as their own.

    Write lib/contracts/locker-attention.test.ts RED first. Cover, at minimum: section ordering; a draft excluded from a non-initiator's view; two viewers of one sheet each seeing their own share; per-party progress counts on a partially-signed sheet; a hidden document omitted for one viewer while remaining in the other viewer's input-derived output; an unknown status landing in the archive.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| viewer session → shared-sheet read | A cross-user read of an agreement whose other parties may have blocked the viewer. |
| viewer session → hide write | A write on a row belonging to a shared legal record. |
| initiator drafts → co-party reads | Unsent working documents that must not become visible on send-adjacent surfaces. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-07 | Information Disclosure | another user's unsent draft appearing in a party's Locker | high | mitigate | Draft rows are filtered to the initiator inside buildAttentionSections and asserted by a dedicated test; the page passes the viewer id and performs no bucketing itself. |
| T-18-08 | Denial of Service | one party destroying a shared legal record from their Locker | critical | mitigate | The hide route issues no delete on any path, is scoped by the caller's user_id, and its test asserts the absence of a delete call rather than merely the presence of an update. |
| T-18-09 | Tampering | the hide write stripping esign evidence and tripping the vault_documents evidence guard | medium | mitigate | document_data is read and merged rather than overwritten; a test asserts pre-existing keys survive. |
| T-18-10 | Information Disclosure | a future audit adding block filtering to the Locker query and destroying a blocked party's access to their own signed agreement | high | mitigate | The exception is documented in-source at the query with its rationale, its narrow scope, and a citation to design section 10c (P18-12). |
| T-18-11 | Information Disclosure | co-party legal name, PRO and IPI shown on a shared agreement | low | accept | Disclosed by design — the parties signed the same document (design section 10b). The bound is that the Locker exposes only this agreement's data and never a party's other catalog data; the attention module receives no such data to leak. |
| T-18-12 | Spoofing | a cross-party action carrying user-supplied text across a block | high | mitigate | This plan adds one cross-party-visible action, hide, and it is self-scoped and emits nothing. No route added here accepts a note, reason or message field (P18-13). |
| T-18-SC | Tampering | npm/pip/cargo installs | low | accept | No package installs in this plan. |
</threat_model>
