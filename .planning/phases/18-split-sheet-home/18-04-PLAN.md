<objective>
Fix the readiness gate that fails in the dangerous direction. Today signedOf('split_sheet') asks whether all split-sheet documents for a project are signed, so a five-track EP with ONE signed sheet has a total of one and a signed count of one and scores complete at 15/15 — four undocumented songs, fully green (18-CONTEXT finding 4, design section 6). This plan replaces that with coverage across the project's tracks, taking the minimum tier across the tracks that need a sheet, implemented identically in the TypeScript derivation and the SQL trigger and anchored to one shared fixture, with the legacy wet-sign path preserved.

Purpose: the readiness score exists to prevent exactly the failure it is currently producing. This is the most consequential finding in the phase and it is live today.
Output: a shared scenario fixture, a pure coverage module, both derivations wired to it, migration 065, and a per-track acknowledgment so honest solo-written songs are not punished.

This plan changes scores users can already see. It carries two blocking checkpoints: one to settle which tracks count toward coverage before anything is built, and one for the developer to sign off on the visible score movement before the migration is applied.
</objective>

<task type="checkpoint:decision" gate="blocking-human">
  <name>Task 1: Decide the coverage denominator</name>
  <action>Stop before building. Put the coverage-denominator question below to the developer and record the answer in the summary; Task 2's tracksNeedingSheet() implements whichever option is chosen.</action>
  <decision>Which of a project's tracks count toward split-sheet coverage?</decision>
  <context>
    Coverage scoring divides covered tracks by tracks needing a sheet, and takes the minimum tier across that set. What goes in the denominator decides how harsh the new score is. Because the rule is a minimum, a single uncounted track holds the whole item at zero — so this choice is not cosmetic.

    A solo-written song arguably needs no split sheet, though best practice says document it anyway. The design leans toward option A: strict, with an explicit per-track acknowledgment that lets an artist say a song was solo-written and no sheet is needed, which prompts the right thought without punishing the honest answer.

    Everything downstream is built to make this a one-predicate change: the denominator is computed by a single named function in lib/vault/readiness-coverage.ts, so a different answer here changes that function and its fixture rows, not the architecture.
  </context>
  <options>
    <option id="strict-with-ack">
      <name>A — Every track needs a sheet, with a per-track solo-written acknowledgment (design's lean)</name>
      <pros>Prompts documentation on every song, which is the behavior the product exists to encourage. The artist keeps an honest escape hatch. The acknowledgment is an explicit, auditable statement rather than an inference.</pros>
      <cons>Requires the artist to act on solo-written tracks before the score can reach complete. Existing projects will read lower until acknowledged.</cons>
    </option>
    <option id="composer-count">
      <name>B — Only tracks whose metadata shows more than one composer need a sheet</name>
      <pros>No new artist action; the denominator derives itself from data already captured in the Metadata Studio.</pros>
      <cons>Depends on composer metadata being complete, which the metadata readiness item exists precisely because it often is not. A track with no composers captured would silently need no sheet — a second failure in the dangerous direction.</cons>
    </option>
    <option id="strict-only">
      <name>C — Every track needs a sheet, no exceptions</name>
      <pros>Simplest rule, nothing to maintain, impossible to game.</pros>
      <cons>Punishes the honest solo writer permanently, with no way to reach a complete score on a legitimately solo-written release.</cons>
    </option>
  </options>
  <resume-signal>Select: strict-with-ack, composer-count, or strict-only.</resume-signal>

    lib/vault/coverage-fixtures.ts exports one scenario table, each row a named project shape — its tracks, their attached sheets and those sheets' statuses, any signed split-sheet documents, any acknowledgments — paired with the expected point value and expected status. This table is the parity anchor: the TypeScript test runs the derivation against it, and the migration test asserts the SQL implements the same rule against the same named scenarios.

    Scenarios the table must contain, at minimum:
    - The finding itself: five tracks, one executed sheet attached to one of them. Expected: NOT complete, and not 15 points. This row is the regression guard for 18-CONTEXT finding 4.
    - All tracks covered by executed sheets. Expected: complete, 15.
    - All tracks covered but one sheet only at the approved tier. Expected: the lower tier, by the minimum rule.
    - No tracks covered at all. Expected: missing, 0.
    - A legacy signed split_sheet vault_documents row present with zero attachments. Expected: complete, 15 — AM-1's universal fallback wins outright and is unaffected by coverage.
    - A project with no tracks at all. Expected: the pre-existing behavior for that case, unchanged.
    - Under the selected policy from Task 1, the case that policy governs: a solo-written acknowledged track, or a single-composer track, or neither.

    lib/vault/readiness-coverage.ts:
    - tracksNeedingSheet(tracks) computes the denominator and is the single point where the Task 1 policy lives.
    - coverageTier(input) returns the point value and status: zero covered is missing at 0; all covered is complete at the minimum tier across the needing set; partial is warning at the minimum tier, which the minimum rule makes 0 whenever any needing track is uncovered.
    - The result additionally carries the covered and needing counts so the surface can say four of five rather than only showing a colour.
    - A track's tier comes from the best sheet attached to it, scored through the existing SPLIT_SHEET_TIER_MAP; the module does not restate those numbers.
    - An empty needing set yields no coverage signal at all, letting the caller fall back to the legacy path rather than treating it as zero — the same degradation posture projectSplitTier() already takes for an empty status list.
    - Pure module: no I/O.

    readinessItemsForProject() split_sheets branch:
    - The legacy signedOf('split_sheet') complete case still returns 15 first, before any coverage work.
    - When attachment data is supplied, coverage replaces the per-project projectSplitTier call.
    - When attachment data is absent, behavior degrades to exactly what it does today, so every existing caller that does not supply it keeps compiling and behaving unchanged — the same optional-input discipline 17-02 used for split_sheets.
