# Phase 9 — UI Design Contract: Rich Member Profile

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Read this first:** `components/profile/ProfileView.tsx` and `app/u/[handle]/page.tsx` already implement ~70% of the locked `user-profile.html` design (banner, avatar, name/verified/pronouns, role badges with lead-role styling, location, tenure, "Open to" chips, Featured spotlight, releases grid with readiness rings, stats sidebar minus placements, live Follow/DM). This spec does **not** re-derive that already-correct 70% — it briefly confirms it, then goes deep on the 9 genuinely new surfaces this phase must design: (1) presence dot, (2) placements stat, (3) avatar/banner upload, (4) roles/open-to/featured edit UI, (5) wired Share button, (6) public "now playing" player, (7) lyrics slide-up panel, (8) public-player overflow menu, (9) visitor more-options menu.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | **none** — bespoke, hand-rolled Tailwind design system, already locked and ~70% implemented. See rationale below. |
| Preset | not applicable |
| Component library | none — plain React + Tailwind utilities mapped 1:1 to `tailwind.config.ts` tokens (`ink`, `card`, `card2`, `lav`, `lavdim`, `brandindigo`, `brandfuchsia`, `money`, `hair`, `hairstrong`) |
| Icon library | inline SVG, Lucide-style (1.7–2px stroke-width, round linecap/linejoin), hand-authored per icon — matches every existing icon in `ProfileView.tsx` / `PlaybackView.tsx` / `FollowButton.tsx` |
| Font | Inter, weights 400–900, via `--font-sans` (already wired in `tailwind.config.ts`) |

**Why no shadcn:** `components.json` does not exist and RESEARCH.md (Standard Stack, HIGH confidence) explicitly concluded zero new libraries are needed for this phase — every capability (Web Share API, clipboard copy, image upload, drop-up panels) is achievable with what's already in the stack. Introducing a component library mid-build would fight the already-shipped, pixel-correct 70% of this exact page and contradict the codebase's consistent hand-rolled-SVG icon convention. This is a deliberate discretionary call, not an oversight — flagging it here for the checker rather than blocking on an interactive shadcn prompt that the project's own research already answered.

## Spacing Scale

This is a **pixel-faithful recreation of a locked hi-fi design** (`docs/design/wave-4-social-layer/`), not a greenfield 8pt system — the governing source is the exact px values already used throughout `app.css` / `user-profile.html` / `artist-profile.html` and already implemented in `ProfileView.tsx`. Declare the nominal scale for new spacing decisions, but do not force-fit existing/locked measurements onto it.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps inside chips/pills |
| sm | 8–9px | Compact gaps (chip internal gap, role-badge gap) |
| md | 14–16px | Default element spacing (card header margin, track-row gap) |
| lg | 24–28px | Card padding, section gaps (`pf-card` = 26px 28px) |
| xl | 34–36px | Page-level horizontal insets (`pf-wrap`, public-player app bar) |
| 2xl | 48–56px | Hero art bottom offset zone, major section breaks |
| 3xl | 64px+ | n/a this phase |

**Exceptions (locked, reuse verbatim — do not "round" these):**
- Avatar: 168px diameter, 5px border, presence-dot pill inset 12px from bottom/right
- Banner: 280px height, 24px bottom-corner radius, "Edit cover" pill inset 18px from top/right
- Public player hero art: 560px height (fixed, per `artist-profile.html`'s portrait canvas)
- Public player app-bar buttons: 42px round, 26px/30px page padding
- Overflow/more-options menu: 12px radius, ~200px min-width, 6px vertical panel padding, 10px/14px row padding
- Lyrics panel drag-handle: 36×4px, 12px from panel top
- Scrub bar: 6px height, 3px radius, 15px knob
- Play button (public player): 78px circle; transport icon gaps: 42px

## New Surface Specifications

### 1. Presence dot (PROFILE-01)

- Visual (already drawn in `user-profile.html`'s `.pf-avatar .live`): absolute-positioned pill, bottom 12px / right 12px of the 168px avatar circle; `--bg` background, 1px `--border-strong`, emerald (`#34D399`) text + 7px dot, 12px/700 label, 4px/9px padding, 999px radius.
- **Locked behavior for Phase 9 (no live infra exists yet — Phase 11 owns Realtime Presence):** build a `<PresenceDot online?: boolean>` component/slot on the avatar, but when `online` is `undefined`/`false`, **render nothing** — omit the pill entirely. Do not show a hardcoded "Online" badge (would be false for everyone) and do not show an "Offline" negative state either. This is the honest default until Phase 11 wires a real signal into the same prop.
- Do NOT subscribe to any Supabase Realtime Presence channel in this phase — that is explicitly Phase 11 scope (PRESENCE-01/02/03).

### 2. Placements stat (PROFILE-06)

- Reuse the existing `Stat` row component/pattern in `ProfileView.tsx` exactly (14.5px/500 key + 24px/800 value, `--border` bottom, 14px/0 padding).
- Insert as the **3rd row**, in this exact order: Followers → Monthly listeners → **Placements landed** → Avg. readiness (matches `user-profile.html` line order).
- Value color: money-gradient text (`mtext` — amber `#F59E0B`→`#F4C77B`), distinct from avg.-readiness's indigo-fuchsia `gtext`.
- Data source: `SELECT COUNT(*) FROM activity_events WHERE profile_id = X AND kind = 'placement'` — computed at request time, never a cached counter (matches the project's notifications convention).
- Add the "Self-reported by artist" caption (see Copywriting Contract) once under the Stats card header, closing the REQUIREMENTS.md Out-of-Scope labeling gap.

### 3. Avatar/banner upload UI (PROFILE-09)

- **Banner:** reuse the exact `.edit-cover` affordance from `user-profile.html` — pill button, 18px inset top-right of the banner, `rgba(10,9,16,.5)` frosted bg + `backdrop-blur(8px)`, `rgba(255,255,255,.18)` border, 13px/600 white label "Edit cover" + 15px pencil icon. Owner-only.
- **Avatar (new affordance, no mockup precedent):** owner-only hover/focus overlay on the 168px circle — `rgba(10,9,16,.55)` scrim + centered 20px white camera icon + "Edit photo" 11px label, fade-in 160ms ease-out, full-circle click target.
- Upload states: idle → native file picker (`accept="image/jpeg,image/png,image/webp"`) → immediate client-side preview (object URL) with a small spinner + "Uploading…" overlay → success (preview becomes live image, brief ≤480ms checkmark pulse) → error (revert to prior image, inline warn-toned error text below the affordance — use amber/warn styling, never rose/destructive).
- Validation copy: see Copywriting Contract ("Image must be JPG, PNG, or WebP" / "Image must be under 10MB").
- Implementation: new `AvatarBannerUpload.tsx` (client component) → POST FormData to a new `app/api/profile/avatar` / `app/api/profile/banner` route (or one route + `type` param) → bucket `vault-assets` (NOT the dead `release-assets` bucket), path `${user.id}/profile/${type}-${Date.now()}.${ext}`.

### 4. Roles / Open-to / Featured edit UI (PROFILE-02, PROFILE-04, PROFILE-05)

**a) Roles editor**
- Reuse `.pf-rolechip` visual exactly (10px radius, `card2` bg, `hairstrong` border, 14px/600 white, 14px indigo icon).
- Preset multi-select from the fixed list (Artist, Producer, Songwriter, Music Supervisor, A&R, Exec), max 6 total roles.
- "+ Add role" dashed chip (matches `.pf-rolechip.add` — `lavdim` text, dashed border) opens an inline custom-title text input, placeholder `e.g. Mixing engineer`, max 40 chars.
- Lead-role control: no drag-and-drop — each non-lead chip shows a small 11px "Set as lead" text action that moves it to array index 0; the current lead chip renders with the existing `.pf-role.lead` gradient-tinted background instead of the action.
- Each non-lead chip gets a small "×" remove control; block removing the last remaining role (min 1 required).

**b) Open-to chips editor**
- Reuse `.pf-openchip` exactly (emerald pill, check icon, 13.5px/600) as a toggle multi-select over the fixed list: Sync licensing, Co-writes, Features, Brand deals.
- Selected = filled emerald pill; unselected = `card2`/outline chip, same shape.
- Empty state: see Copywriting Contract — omit the section entirely when none selected, no negative-state copy.

**c) Featured-project picker**
- Owner-only ghost button near the Featured section header: `.pf-btn.ghost` visual (card bg, hairstrong border, 15px/700 white, 11px radius), label "Change featured release" (or "Pin a release" if none set yet).
- Opens a list pre-filtered client-side to the owner's **public** releases only — never show private drafts (Pitfall 4). Row: 40px cover thumb + title + type/year.
- Empty state + error copy: see Copywriting Contract.

### 5. Share button — wired (PROFILE-08, D-04–D-07)

- Visual: reuse the existing dead-stub `.pf-btn.ghost` markup in `ProfileView.tsx` (~line 214) verbatim — wire `onClick` only, no restyle.
- Behavior: `onClick` calls `navigator.share({ title: caption, url })` **synchronously as the first statement** (no leading `await` — Safari rejects Web Share calls after the user-gesture window expires). Catch `AbortError` silently (user cancelled). Any other failure or missing `navigator.share` → `navigator.clipboard.writeText('${caption} → ${url}')` → swap the button's own label to "Link copied!" for 1500ms (reuse the existing `PitchCard.tsx` label-swap convention verbatim — no toast library).
- Captions: see Copywriting Contract (track-share vs. profile-share captions differ).
- Visibility gate (D-07): the Share affordance must be **omitted server-side from the rendered markup** for visitors when `artist_profiles.allow_resharing = false` — never CSS-hide client-side (a determined visitor could re-enable it via devtools). The owner always sees their own Share button regardless of this toggle.
- New settings toggle (owner-only): "Allow others to share my music" — plain on/off switch, no confirmation dialog (fully reversible, non-destructive).

### 6. Public "now playing" player — `PublicPlaybackView.tsx` (D-01, D-02, D-03)

Pixel spec sourced directly from `artist-profile.html` (portrait 820×952 canvas):

- **App bar:** absolute, top 0, full width, 26px/30px padding. Left: 42px frosted round back-chevron button. Center: "NOW PLAYING" 14px/700 uppercase, letter-spacing .18em, `rgba(255,255,255,.85)`. Right: 42px frosted round overflow (⋯) button (opens item 8's menu).
- **Hero art:** 560px tall, full-bleed `object-fit: cover`, `object-position: center 32%`. Scrim: `linear-gradient(180deg, rgba(8,7,13,.45) 0%, transparent 26%, transparent 50%, rgba(8,7,13,.6) 82%, #08070d 100%)`.
- **Meta block** (bottom-left over the scrim, 34px side inset, 26px bottom inset): frosted "FUNŪN · Artist profile" pill → eyebrow (release type, 14px/700 uppercase, .2em tracking) → 58px/900 track title, line-height 1 → artist row (30px avatar + 19px/600 name + 19px indigo verified check + 15px lavdim "· N monthly listeners").
- **Player:** 6px scrub bar (3px radius, `rgba(199,203,247,.16)` track, `--grad` fill with a 15px white knob) → times row (14px lavdim, tabular-nums) → transport row (42px gaps: shuffle · skip-prev · 78px gradient play circle with `0 18px 44px -12px rgba(217,70,239,.65)` shadow · skip-next · repeat).
- **Up-next:** "More from {Artist}" 13px/700 uppercase label + "See all" 14px/600 indigo link → track rows (48px rounded-10px cover thumb + 17px/600 title + 14px lavdim "type · year" + 14px lavdim duration). Currently-playing row: title renders in `gtext` gradient style + small indigo equalizer glyph replacing the duration.
- **Data:** reuse `toTrackViews()`; stream only `tracks.audio_file_url` via the existing `is_public` gate + `createSignedUrls()` 2-hour signed-URL pattern. Never read `tracks.metadata.master`.
- **Explicit exclusions (D-01):** no Master/Stems toggle, no waveform, no ISRC/ISWC/BPM table, no split percentages, no file-management/upload affordance. This is a brand-new sibling component — do not fork `PlaybackView.tsx` with conditionals.

### 7. Lyrics slide-up panel — `LyricsPanel.tsx` (D-08, D-09, D-13)

- Trigger: "View lyrics" item in the overflow menu (item 8) — the menu item itself is omitted entirely when `tracks.metadata.lyrics` is empty/null (D-09).
- Visual: bottom sheet covering roughly the lower 70–80% of the player viewport. `--card` (#0E0D1E) surface, 18px top-corner radius, `--border-strong` top border, centered 36×4px drag-handle 12px from the sheet's top.
- Header: 20px/800 track title + a 42px frosted round chevron-down/close button (top-right of the sheet, matching the app-bar button style).
- Body: 16px/500, line-height 1.6, `--lav` text, left-aligned, scrollable on overflow.
- **Player keeps running underneath** — this is an overlay, not a route change and not a modal that pauses audio.
- Motion: slide up/down at 280ms ease-out (within the locked 160–480ms app.css budget), no bounce; respect `prefers-reduced-motion` (cross-fade instead of slide).
- Dismiss: drag-handle swipe/tap, the close button, or tapping the scrim outside the sheet.
- D-13 scope: render only `lyrics.text` (static). If `lyrics.synced` exists, ignore it this phase — no line-highlight logic built now.

### 8. Overflow (⋯) menu — public player (D-14)

- Trigger: the app bar's existing overflow button (item 6). Click-outside-to-close dropdown (reuse the `mousedown`-listener pattern from `CollaboratorPicker.tsx`).
- Panel: anchored below-right of the trigger, `--card` bg, 1px `--border`, 12px radius, ~200px min-width, `0 12px 30px -10px rgba(0,0,0,.5)` shadow, 6px vertical padding.
- Items, in this exact order: **View credits** → **View lyrics** (omitted if no lyrics) → **Copy link**. Each row: 15px/600 white + 18px leading icon, 10px/14px padding, `--card2` hover bg.
- "View credits" opens a lightweight names+roles list from `readComposers()` — reuse `PlaybackView.tsx`'s credits-row visual (avatar/initials + name + role) but **strip the split-percentage element entirely** (D-11 — no `%` anywhere on the public surface).
- "Copy link" triggers the clipboard-fallback half of the Share mechanism directly (item 5) — same "Link copied!" confirmation, shown as a brief inline note near the menu (the trigger here is icon-only, so there's no button label to swap).

### 9. Visitor "more-options" (⋯) menu — profile page (PROFILE-08)

- Trigger: the existing `.pf-btn.ghost.icon` circular ⋯ button already drawn next to Follow/Message in `user-profile.html` — build the dropdown container now.
- Panel: identical visual language to item 8 (`--card`, `--border`, 12px radius, click-outside-to-close).
- **Phase 9 contents: "Copy profile link" only.** Uses the same Web-Share-then-clipboard-fallback mechanism as item 5.
- **Explicit exclusion:** do NOT add "Report" or "Block" items in this phase (Phase 13 / SAFETY-01/02) — a visually-present-but-non-functional stub violates this project's "no silent no-ops" error-handling convention (CLAUDE.md). Leave a code comment marking the Phase 13 insertion point; ship with exactly one menu item.
