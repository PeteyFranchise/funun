<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/profile/avatar | Untrusted multipart upload crosses into Supabase Storage; MIME + size + path-scoping are the gate |
| Storage object path → artist_profiles column | The uploaded public URL is written to avatar_url/banner_url on the authenticated user's own row |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-05 | Tampering / Elevation of Privilege | Uploaded file smuggling a non-image payload (MIME-sniffing bypass) | high | mitigate | `EXT_BY_MIME` allowlist keyed off `file.type` (image/png,jpeg,webp only); 10MB `MAX_BYTES` cap; client-declared type used only for bucket-path naming, never trusted for content — mirrors the live assets-route pattern |
| T-09-06 | Elevation of Privilege | User uploads into another user's storage folder | high | mitigate | Path is `${user.id}/profile/...` where `user.id` comes from `auth.getUser()` (server-derived, not client-supplied); existing vault-assets RLS policy `(storage.foldername(name))[1] = auth.uid()::text` enforces the first-segment ownership at the Storage layer |
| T-09-07 | Tampering | Client writes avatar_url/banner_url to a column via a widened API | low | mitigate | Write-back updates only the two image columns on `.eq('id', user.id)`; the broader profile PATCH allowlist (Plan 01) still gates any client-driven column writes |
</threat_model>
