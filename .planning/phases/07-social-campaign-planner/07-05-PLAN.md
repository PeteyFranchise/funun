## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client component → campaigns API | These components issue the fetch/PATCH calls; all authorization is server-side (07-03/07-04), the client is untrusted |
| optimistic local state → server truth | A failed PATCH could leave the UI showing a state the server rejected |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-17 | Tampering (client-trust) | CampaignCalendar PATCH bodies | medium | mitigate | Client sends only single-field slot edits; the server (07-03) re-validates via sanitizeSlotEdit and re-derives ownership — no authorization logic lives in the client component |
| T-07-18 | Repudiation (stale UI) | optimistic update rollback | low | mitigate | On PATCH failure the component re-fetches authoritative server state (LaunchpadRoom WR-02 pattern) rather than blind-reverting, so the UI cannot silently diverge from the server after a rejected write |
| T-07-SC | Tampering | npm installs | low | accept | No new packages (RESEARCH.md audit) — components use only existing React + Tailwind |
</threat_model>
