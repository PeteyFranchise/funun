## Interaction Contracts (cross-component, not purely visual â€” required for the checker's Dimension 2 pass)

- **Checkbox-independent-of-panel:** `CampaignSlot`'s completion checkbox must call `e.stopPropagation()` exactly like `ChecklistItem.tsx`, so toggling completion never triggers inline-edit mode or opens `SlotGeneratePanel` (D-13, direct mirror requirement).
- **Preview-then-accept, never silent overwrite:** both `SlotGeneratePanel` (D-10) and `SaveToCalendarPicker` (D-11) must show the AI output before any write occurs; the write is a distinct user action ("Use this" / "Save to slot"), never an automatic side effect of generation completing.
- **Optimistic update + re-fetch-on-failure rollback:** every PATCH-backed interaction in this phase (completion toggle, inline caption edit, posting-time override, platform selection change) follows `LaunchpadRoom.tsx`'s exact pattern â€” update local state immediately, PATCH, and on failure re-fetch authoritative server state rather than blindly reverting to the pre-optimistic snapshot (avoids the race-condition class Phase 5's WR-02 fixed).
- **Escape/backdrop dismissal:** `SlotGeneratePanel` (slide-in shell) supports `Escape` and backdrop-click close, matching `TipPanel`. `SaveToCalendarPicker` (centered modal shell) supports backdrop-click close at minimum; `Escape` support is consistent with the panel and should be added for parity even though RESEARCH.md doesn't explicitly require it for the modal variant.
- **Destructive action requires inline confirm, never a bare click-to-delete:** `CampaignHistoryList`'s hard-delete (D-05) always routes through the inline confirm block â€” no single-click delete anywhere in this phase, matching the `ChecklistAdmin.tsx` precedent this project has already established for hard-delete actions.
