<objective>
Build the campaign lifecycle API: `campaigns/route.ts` (POST generate a 4-week calendar, GET list, PATCH set-active, DELETE an inactive campaign) and `slots/[slotId]/route.ts` (PATCH a single slot's caption / posting_time / completed). Both routes copy `app/api/tools/[slug]/route.ts`'s auth + user-scoped fetch + batch-Anthropic + `extractJson` shape but write to `social_campaigns` instead of `tool_outputs`, and enforce D-04's one-active-per-project invariant with a same-request flip on top of migration 033's DB-level backstop.

Purpose: This is the server backbone the whole feature turns on — generation, campaign switching/deletion, and every slot mutation flow through these two files. The IDOR guard (load parent campaign scoped by user_id before touching any slot) and the sanitizeSlotEdit allowlist are the security-critical parts.
Output: two new route files.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → campaigns/slots API | Authenticated artist requests; middleware does NOT cover /api, so each route is its own auth boundary |
| slotId param → posts[] mutation | A caller-supplied slotId/campaignId could reference another user's campaign (IDOR) |
| request body → posts JSONB | A malicious body could try to overwrite the whole posts array |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-07 | Elevation of Privilege (IDOR) | slots/[slotId] PATCH | high | mitigate | Load parent campaign scoped by `.eq('user_id', user.id).eq('id', campaignId)` and 404 before touching posts[] — ownership re-derived from the parent row, a bare slotId is never trusted (RESEARCH.md Known Threat Patterns) |
| T-07-08 | Tampering | posts array overwrite | high | mitigate | Every PATCH loads current posts server-side, applies only sanitizeSlotEdit()'s allowlisted fields to the located slot, re-saves — a client-supplied full posts array is never accepted as the column value (RESEARCH.md Anti-Patterns) |
| T-07-09 | Spoofing | missing per-route auth | high | mitigate | Every handler independently calls `auth.getUser()` and 401s — no reliance on middleware, which excludes /api (RESEARCH.md V2) |
| T-07-10 | Tampering (data integrity) | concurrent set-active | high | mitigate | API flip-old-off-then-set-new inside one request, layered on migration 033's partial unique index — the DB rejects a second active row even if two requests race (Pitfall 5) |
| T-07-11 | Tampering | AI output corrupting posts | medium | mitigate | POST routes model output through readCalendarPosts()→readPosts() enum/range validation before insert — hallucinated platform/content_type/week dropped, not stored |
</threat_model>
