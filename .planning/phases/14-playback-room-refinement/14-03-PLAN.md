<objective>
Build the two JSON-only metadata routes that persist a stems ZIP reference and an instrumental audio reference onto the canonical track row after the browser has uploaded the bytes directly to Supabase Storage (D-05/D-07). These routes receive `{ path, size, name/ext }` — never file bytes — so they stay far under Vercel's 4.5MB body ceiling (Pitfall 1). Each route supports POST (persist ref) and DELETE (remove object + clear ref).

Purpose: The direct-to-storage upload (Plan 05) writes bytes to Storage but cannot write to the `tracks` table from the browser — a small authenticated server call must persist the reference. Making this a JSON-only route (not a byte proxy) is the whole architectural point of the phase's upload design.
Output: Two new API route files.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → metadata route (client-declared path/size/name) | The client posts the storage path it just uploaded to; the route persists it onto the track row |
| metadata route → Supabase (service client for DELETE storage.remove) | DELETE uses the service role to remove the object; owner is verified first |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-14-04 | Elevation of Privilege | stems/instrumental POST+DELETE (non-owner invocation) | high | mitigate | Both handlers fetch the track with `.eq('user_id', user.id)` and return 404 before any read/write — identical to the existing audio route's owner gate (ASVS V4, RESEARCH §Security Domain). Verified by node grep + manual two-session check. |
| T-14-05 | Tampering | client-declared storage path in POST body | medium | mitigate | The bucket's RLS policy already constrained the ORIGINAL upload to `(storage.foldername(name))[1] = auth.uid()::text` (migration 004), so a malicious `../otherUser/...` path could never have been written by this user's session in the first place; the metadata route persists a reference to an object the RLS already proved is owner-scoped. Route additionally never lets the client name an arbitrary bucket. |
| T-14-06 | Tampering | spoofed non-ZIP with .zip MIME to stems slot | low | accept | The metadata route does not (and by design cannot) validate bytes — it never sees them. The authoritative check is the bucket's `allowed_mime_types` allowlist (Plan 01), enforced by Storage at upload time. Accept rationale: byte-level validation belongs at the storage boundary, not this JSON route. |
</threat_model>
