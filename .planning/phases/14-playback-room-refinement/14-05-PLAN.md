<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Supabase Storage (direct TUS upload) | The browser uploads stems/instrumental bytes straight to Storage using the user's own session token; bucket RLS constrains the write path to the user's own folder |
| play page (server) → Supabase Storage (service client signing) | The owner-gated server component mints short-lived signed URLs for the owner's own files only |
| client → /stems, /instrumental routes | Covered by Plan 03's owner gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-14-09 | Tampering / Elevation of Privilege | direct-to-storage upload path (client-chosen object path) | high | mitigate | The client always derives the upload path from the authenticated user's own id (`${userId}/${projectId}/${trackId}...`) and the bucket RLS policy (migration 004) enforces `(storage.foldername(name))[1] = auth.uid()::text` — a `../otherUser/...` path is rejected by Storage. The client never accepts a path from user input (RESEARCH §Security Domain). |
| T-14-10 | Information Disclosure | signed URLs for master/instrumental/stems on the play page | medium | mitigate | URLs are minted server-side only on the owner-gated page (`.eq('user_id', ...)`), at a 2-hour TTL (existing precedent), from the service client — a non-owner rendering the page hits `notFound()` before any URL is minted. |
| T-14-11 | Denial of Service | 250MB upload from the browser | low | accept | Bucket `file_size_limit` (250MB, Plan 01) caps the object size at the storage layer; TUS chunking bounds client memory. Accept: abuse beyond the per-file cap is out of scope for v1 and gated by owner auth. |
| T-14-SC | Tampering | (see Plan 01 T-14-SC — tus-js-client provenance) | high | mitigate | Package legitimacy already gated at Plan 01's blocking human checkpoint. |
</threat_model>
