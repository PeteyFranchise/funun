<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → mutation routes | follow target, wall/endorsement profile_id, comment project_id are untrusted input (each route's existing RLS already gates the primary write) |
| route → service-role notification insert | cross-user write; recipient is derived server-side (followeeId / profileId / resolved project owner), actor snapshot from the caller's own profile |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-10 | Spoofing | notification actor identity across the 4 routes | medium | mitigate | Actor snapshot is read from the caller's own `artist_profiles` row keyed by `auth.uid()`; recipient is derived server-side from the mutation's own columns — no client-supplied identity is trusted |
| T-10-11 | Information Disclosure | release_comment notification revealing a private project's owner/title | low | mitigate | The owner lookup uses the route's existing session-bound access; the release_comments insert already passed RLS (commenter had visibility), so no new disclosure surface is opened; notification body carries only the title the commenter already saw |
| T-10-12 | Denial of Service | a notification-insert failure aborting the primary social mutation | medium | mitigate | Every notify is wrapped in try/catch AFTER the primary mutation (RESEARCH Pattern 1) — the follow/post/endorse/comment always succeeds even if notification delivery fails |
| T-10-SC | Tampering | npm/pip/cargo installs | high | accept | No packages installed; no install task exists |
</threat_model>
