<objective>
Build the notification bell + dropdown panel and the net-new authenticated header row that hosts them. Per RESEARCH Pitfall 4, no topbar exists in the authenticated app shell today — `app/(artist)/layout.tsx` renders only `<ArtistNav>` (left sidebar) beside `{children}` — so this plan adds a slim sticky header row inside the content column and mounts `<NotificationBell />` there exactly once. The bell subscribes to Supabase Realtime globally (D-13, unlike `DmWidget`'s panel-gated subscription), with a stable per-user channel name and proper cleanup (Pitfall 5), and a ~25s poll reconcile fallback. The panel (D-08–D-11) lists recent notifications, marks all read on explicit click only (D-09), acts on connection-request rows inline (D-10), and cursor-paginates older notifications on scroll (D-11).

Purpose: This is the first UI ever built on top of the `notifications` table (antenna_match/application_received rows have been silently accumulating with no viewer). It reuses the bespoke Tailwind design system verbatim per UI-SPEC — no new packages, hand-rolled SVG icons, the inline `timeAgo()` helper from `Wall.tsx` (not `date-fns`).
Output: two new client components + a modified layout.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser Realtime channel → notifications rows | The client-supplied `filter: user_id=eq.<uid>` is a convenience filter, NOT a security boundary |
| panel inline actions → connections PATCH | connectionId comes from the notification row's data; the PATCH is RLS-gated server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-13 | Information Disclosure | Realtime channel used as an unauthenticated read path | medium | mitigate | Supabase Realtime respects RLS on postgres_changes for authenticated clients; the `user_id=eq.${userId}` filter is paired with the notifications RLS (`USING auth.uid() = user_id`, migration 009), never relied on as the sole access control (RESEARCH Security Domain) |
| T-10-14 | Denial of Service | Realtime channel leakage (TooManyChannels) across navigations/tabs | high | mitigate | Stable per-user channel name (`notifications-${userId}`) so tabs reuse the channel; always-return `removeChannel` cleanup; memoized client (Pitfall 5, corroborated by Supabase troubleshooting docs) |
| T-10-15 | Information Disclosure | notification title/body rendered in the panel (XSS) | low | mitigate | Titles/bodies render as plain React text (JSX auto-escapes); no `dangerouslySetInnerHTML` anywhere in this component |
| T-10-16 | Tampering | client-incremented unread badge drifting from truth | low | mitigate | Badge count is always set from a fresh COUNT fetch, never client-incremented (STATE.md convention, Anti-Patterns guard) |
| T-10-SC | Tampering | npm/pip/cargo installs | high | accept | No packages installed (bespoke Tailwind + hand-rolled SVG + copied timeAgo, per UI-SPEC + RESEARCH Package Legitimacy Audit); no install task exists |
</threat_model>
