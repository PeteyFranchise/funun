<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → connections/notifications routes | connectionId, action, note, before-cursor are untrusted input |
| route → service-role notification insert | cross-user write; the notification recipient is derived server-side from the connection row, never from client input |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-06 | Tampering / Elevation of Privilege | connect status transition (accept/decline/withdraw) | high | mitigate | Status UPDATE uses the SESSION client only; RLS's two-policy split (migration 035) enforces addressee-accepts / requester-withdraws — the route never uses a service-role client for the transition, so a requester cannot self-accept |
| T-10-07 | Spoofing | notification recipient / actor identity | medium | mitigate | Recipient (requester_id / addressee_id) is read server-side from the connection row; actor snapshot is read from the caller's own `artist_profiles` row keyed by `auth.uid()` — client cannot forge who a notification is from or to |
| T-10-08 | Information Disclosure | reading another user's notifications via the GET route | medium | mitigate | GET scopes every query to `user_id = user.id` (session-derived), backstopped by existing notifications RLS `USING auth.uid() = user_id` (migration 009) — no client-supplied user id is trusted |
| T-10-09 | Denial of Service | offset-pagination skip/duplicate under concurrent inserts | low | mitigate | created_at-cursor pagination (RESEARCH Open Question #2) is race-safe on a table that receives concurrent inserts by design |
| T-10-SC | Tampering | npm/pip/cargo installs | high | accept | No packages installed; no install task exists |
</threat_model>
