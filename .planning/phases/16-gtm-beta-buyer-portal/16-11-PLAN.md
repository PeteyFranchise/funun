<objective>
Give artists a complete, explained home for the industry identifiers attached to their music: inline explainers on every identifier field, the missing DDEX-level identifiers added at their correct levels, and a cross-project code sheet they can export and keep on file.

Purpose: Funūn already captures more registration-grade identifiers than the sync competitors (ISRC, ISWC, UPC, IPI, ISNI, IPN, MLC ID) and validates them properly — the ISWC check digit is computed, not regex-matched. But the UI assumes the artist already knows what each code is and how to get one, every export is scoped to a single project, and three DDEX identifiers have no home at all. This plan closes those three gaps.
Output: The identifier guide module, migration 066, the ERN emission fixes, the code-sheet export, the reference page, and the Metadata Studio popovers.
</objective>

  <name>Task 1: Identifier guide module — content + DDEX level map</name>
  <files>lib/metadata/identifier-guide.ts, lib/metadata/identifier-guide.test.ts</files>
  <behavior>
    IDENTIFIER_GUIDE:
    - every identifier entry carries a non-empty id, label, ddexLevel, identifies, issuedBy, howToGet, unlocks, and officialUrl.
    - ddexLevel is one of 'party' | 'work' | 'resource' | 'release'.
    - officialUrl is https and points at the issuing body's own domain, never a Funūn route.
    - getIdentifiersForLevel('release') returns exactly the release-level entries.
    - every identifier Funūn persists has a guide entry (test asserts the stored-field list and the guide keys stay in sync, so adding a stored identifier without an explainer fails CI).
  </behavior>
  <action>
    Create lib/metadata/identifier-guide.ts as a pure, client-safe content module (no Node deps, no imports from route code), following the LABELS/VALUES convention already used in schema.ts.

    Type: `IdentifierGuideEntry = { id, label, ddexLevel: DdexLevel, identifies, issuedBy, howToGet, unlocks, formatExample, officialUrl, storedAt }` where `storedAt` names the column/JSONB path Funūn persists it to (or null for informational-only entries).

    Cover every identifier below, grouped by DDEX level. Content must be factual and process-oriented:

    PARTY level (identifies a person or company):
    - **IPI / CAE** — identifies a songwriter or publisher. Issued by CISAC through the writer's PRO. Obtained automatically when joining a PRO. Unlocks correct writer attribution on PRO and MLC registrations. Stored: Composer.ipi (per writer) and artist_profiles.ipi.
    - **ISNI** — identifies any public party (recording artist, label, publisher). Issued by the ISNI International Agency via a registration agency; many artists receive one indirectly through a distributor or library. Unlocks disambiguation between same-named artists across databases. Stored: Performer.isni, and (new, Task 2) artist_profiles.isni.
    - **IPN** — identifies a performer for neighbouring rights. Issued by the performer's collecting society (e.g. SoundExchange/PPL/GVL region equivalents). Unlocks non-featured/featured performer royalty attribution. Stored: Performer.ipn.
    - **MLC Member ID** — identifies a rightsholder at The MLC (US mechanical). Obtained by registering as a member at themlc.com. Stored: artist_profiles.mlc_id.
    - **DPID (DDEX Party ID)** — identifies a company inside DDEX message exchange. Issued by DDEX to members. Note honestly that this is a label/distributor-level identifier most independent artists never hold personally; Funūn reads it from the DDEX_DPID env var and falls back to a documented placeholder in ERN exports. Stored: environment config, not artist data.

    WORK level (identifies a composition):
    - **ISWC** — identifies the underlying song as written, independent of any recording of it. NOT self-assignable: allocated centrally by CISAC and returned through the writer's PRO after the work is registered. Unlocks performance (PRO) and mechanical (MLC) royalty matching. Stored: tracks.iswc.

    RESOURCE level (identifies a specific recording):
    - **ISRC** — identifies one specific recording of a song; a remix, live version, or re-record each need their own. Self-assignable once the artist holds a registrant code from their national ISRC agency (in the US, the RIAA via usisrc.org); Funūn then mints sequential codes from that registrant code. Unlocks streaming analytics, SoundExchange digital-performance royalties, and YouTube Content ID matching. Stored: tracks.isrc.

    RELEASE level (identifies the product):
    - **UPC / EAN (GTIN)** — identifies the release as a retail product. Usually supplied free by the distributor at delivery; can also be bought directly from GS1. Unlocks store/DSP shelf identity and chart reporting. Stored: vault_projects.upc, emitted as ICPN in ERN.
    - **GRid** — Global Release Identifier, the digital-release counterpart to UPC, issued by IFPI/DDEX and typically supplied by a distributor. Stored: vault_projects.grid (new, Task 2).
    - **Catalog number** — the label's own internal release number. Self-assigned; no issuing body. Stored: vault_projects.catalog_number (new, Task 2).

    Also include an informational-only entry for **ISAN** (audiovisual works), marked storedAt: null, since sync buyers licensing to film/TV may encounter it — flagged as context, not something Funūn tracks.

    COPY CONSTRAINT: explain what each body does and how the process works, and link out. Do NOT recommend which PRO, society, or distributor to choose — that is a consequential personal decision, and app/(artist)/vault/[projectId]/rights/page.tsx already sets the correct precedent of describing bodies neutrally and linking out. Do not state fees or turnaround times; those change and would go stale silently.

    Write identifier-guide.test.ts to the behavior block first (RED), including the sync test that fails when a stored identifier lacks a guide entry.
  </action>
  <verify>
    <automated>npx jest lib/metadata/identifier-guide.test.ts && npx tsc --noEmit && npm run lint</automated>
  </verify>
  <done>Every stored identifier has a complete, level-tagged explainer; the sync test guards against future drift; no recommendation language present.</done>
</task>
