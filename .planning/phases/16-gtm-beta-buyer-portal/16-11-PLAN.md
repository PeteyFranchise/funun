<objective>
Give artists a complete, explained home for the industry identifiers attached to their music: inline explainers on every identifier field, the missing DDEX-level identifiers added at their correct levels, and a cross-project code sheet they can export and keep on file.

Purpose: Funūn already captures more registration-grade identifiers than the sync competitors (ISRC, ISWC, UPC, IPI, ISNI, IPN, MLC ID) and validates them properly — the ISWC check digit is computed, not regex-matched. But the UI assumes the artist already knows what each code is and how to get one, every export is scoped to a single project, and three DDEX identifiers have no home at all. This plan closes those three gaps.
Output: The identifier guide module, migration 066, the ERN emission fixes, the code-sheet export, the reference page, and the Metadata Studio popovers.
</objective>

  <name>Task 4b: Generalized identifier generator (catalog number, GRid, UPC)</name>
  <files>lib/metadata/generate.ts, lib/metadata/generate.test.ts, app/api/metadata/generate-identifier/route.ts</files>
  <behavior>
    Three assignment modes are in play (per D-16f):
      • platform_issued  → 'grid' only: minted under Funūn's OWN issuer code, held in server config, from a single GLOBAL counter shared across all artists.
      • self_assign_with_prefix → 'upc', 'catalog_number', 'isrc': minted from a prefix on the ARTIST's own profile, from that artist's per-scheme counter.
      • centrally_allocated → 'iswc','ipi','isni','ipn','mlc_id','dpid': never generated.
    canGenerate(scheme, profile, platformConfig):
    - 'grid' → eligible when Funūn's issuer code is configured (platformConfig), UNLESS the artist/label has supplied their own issuer code on the profile, in which case theirs is used instead. Never depends on an artist-held GRid prefix, because the default path is platform-issued.
    - 'upc' → not eligible unless the profile has the artist's own gs1_company_prefix; Funūn holds none, so absent a personal prefix the answer is always no. Reason names the missing prefix AND states Funūn does not issue UPCs.
    - 'catalog_number' with no catalog_number_prefix → not eligible, with reason.
    - any centrally-allocated scheme → not eligible, ALWAYS, regardless of profile or platform config; reason names the issuing body.
    generateIdentifier(scheme, artistProfile, artistCounters, platformState):
    - 'grid' produces scheme 'A1' + 5-char issuer code + 10-char release number + check character, 18 alphanumeric total (hyphens display-only). The issuer code is Funūn's platform code by default, or the artist's own if present. The 10-char release number is drawn from the counter that OWNS that issuer code: the GLOBAL platform counter for Funūn's code, or the artist's counter for an artist-owned code. Getting this wrong (drawing a platform-issued release number from a per-artist counter) collides two artists' releases under Funūn's code — the whole correctness risk of this mode.
    - a generated GRid's check character validates under ISO 7064:1983 Mod 37,36 — test against the published example A1-2425G-ABC1234002-M.
    - 'upc' produces a 12-digit GTIN: the artist's company prefix + sequential item reference + computed mod-10 check digit; passes the existing isValidUpc().
    - 'catalog_number' produces prefix + zero-padded sequence (e.g. FUN-0007); no check digit.
    - the returned counter state increments only the counter that owns the minted code (global vs per-artist), leaving all others untouched.
    - exhausting the space under a code returns null rather than wrapping or colliding.
    - generating never mutates input arguments (pure function; caller persists).
  </behavior>
  <action>
    Build lib/metadata/generate.ts as the generalized "prefix + counter + check digit" utility that lib/metadata/identifiers.ts already implements once for ISRC. Read identifiers.ts first — `nextDesignation`, `currentIsrcYear`, and `formatIsrc` are the working precedent, and the ISRC path must keep behaving exactly as it does today. Either refactor ISRC onto the shared utility or leave it in place and mirror its shape; do NOT create a second competing counter convention, and do not change any ISRC output format.

    THE GLOBAL-COUNTER RULE (platform_issued, D-16f): a GRid minted under Funūn's own issuer code draws its release number from a SINGLE platform-wide counter, not a per-artist one. Two artists generating a GRid on the same day must receive different release numbers. Model the platform counter as server-owned state (a dedicated single-row table or a keyed config row updated under a row lock / atomic increment), NOT a value on any artist_profiles row, and NOT a value the client can supply. This is the one place in the plan where the counter is shared across users, and a per-artist counter here is a latent collision bug that unit tests on a single artist will not catch — so include an explicit test that two sequential platform-issued mints yield distinct, non-decreasing release numbers.

    Implement the check digits with their CORRECT, DIFFERENT algorithms — this is the most likely place for a wrong assumption in this plan, because three identifiers in the same module use three different schemes:
    - UPC: GS1 mod-10 (weighted 3-1 alternating over 11 digits).
    - GRid: **ISO 7064:1983 Mod 37,36** over the 17 preceding characters, producing an alphanumeric check character — NOT a digit, and NOT the same algorithm as either of the others.
    - ISWC: the existing `iswcCheckDigit` in identifiers.ts, unchanged.
    Do NOT reuse `iswcCheckDigit` for UPC or GRid. Verify each against a published reference value in the tests rather than trusting the implementation to be self-consistent.

    THE CENTRAL GUARDRAIL — eligibility is structural, not advisory: `generateIdentifier` must be unreachable for any scheme whose guide entry is `centrally_allocated`, and unreachable for a self-assignable scheme whose required prefix (artist-held for UPC/catalog, platform-or-artist-held for GRid) is absent. There is no force flag, no override parameter, and no "generate anyway" path. A caller cannot opt out, because the check lives inside the generator rather than in the UI. This is what prevents Funūn from ever emitting a UPC under a GS1 prefix nobody owns.

    THE DISTRIBUTOR-GRID CONFLICT RULE: if a release already carries a GRid (entered manually / imported from the distributor), the generator must refuse to mint a second one — one release, one GRid. Minting is offered only when the release's GRid field is empty; an existing value, whatever its provenance, blocks generation with a reason.

    POST /api/metadata/generate-identifier accepts a scheme and target (project or track), re-runs `canGenerate` server-side against the caller's own profile AND the platform config — never trusting an eligibility result from the client — mints the code, persists it together with its provenance marker ('generated'), and atomically increments the owning counter (global platform counter for a platform-issued GRid, the artist's counter otherwise). Guard against double-minting from a double-clicked button: increment and write in one operation so a retry cannot burn or reuse a number. Scope writes strictly to the caller's own project/track, but note the platform GRid counter is shared infrastructure the caller advances but does not own.

    Write generate.test.ts to the behavior block first (RED), then implement. Include an explicit test that every centrally-allocated scheme is refused even when the profile is fully populated.
