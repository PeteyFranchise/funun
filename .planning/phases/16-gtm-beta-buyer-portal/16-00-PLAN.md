<objective>
Add artist-authored mood/descriptor tagging, energy level, and a vocal/instrumental indicator to the Metadata Studio, stored in tracks.metadata JSONB.

Purpose: This is the one metadata dimension a sync supervisor uses before genre, and Funūn has no per-track version of it today. `mood_tags` exists only on `artist_profiles.sound_identity` (artist-level, benchmark-derived, not artist-authored) and on `antenna_opportunities` (the demand side's ask) — nothing describes an actual song. Plan 16-05 specifies buyer catalog filters for "mood/energy" and "vocals" that currently have no backing data; this plan creates it. Value extends past the buyer portal: Antenna matching today compares opportunity mood tags against artist-level analysis rather than the songs themselves.
Output: The descriptor vocabulary, types, read/sanitize helpers with tests, PATCH allowlist support, and the Metadata Studio editing UI.
</objective>

<task type="auto" tdd="true">
  <name>Task 1: Descriptor vocabulary, types, and read/sanitize helpers</name>
  <files>lib/metadata/schema.ts, lib/metadata/descriptors.test.ts</files>
  <behavior>
    sanitizeDescriptors(input):
    - unknown/non-array moods → empty array, never throws.
    - a mood term not in the controlled vocabulary → dropped silently (not persisted as free text).
    - duplicate mood terms → de-duplicated.
    - more than MOODS_MAX terms → truncated to MOODS_MAX.
    - energy not in ENERGY_VALUES → null.
    - vocal not in VOCAL_VALUES → null.
    - fully empty/absent input → null (meaning "untagged"), not an empty object.
    readDescriptors(metadata):
    - metadata without a descriptors key → null.
    - descriptors present but containing terms retired from the vocabulary → those terms filtered out on read, remaining terms preserved.
  </behavior>
  <action>
    Add to lib/metadata/schema.ts, following the exact shape of the existing PRO / COMPOSER_ROLE / PERFORMER_ROLE triplets (type + LABELS record + VALUES array):

    - `Mood` type + `MOOD_LABELS` + `MOOD_VALUES`. Build the term list by reading what the demand side already uses: check `antenna_opportunities.mood_tags` usage and the `SoundIdentity.mood_tags` shape in types/index.ts, and any existing mood constants in lib/matching or lib/benchmarks. Reuse those terms as the seed vocabulary so supply and demand speak the same language — if a constant list already exists, import/extend it rather than defining a second competing list, and say so in a comment. Aim for a supervisor-usable set (roughly 20-40 terms spanning emotional tone, e.g. uplifting/melancholy/tense/hopeful/aggressive/serene, and cinematic register, e.g. epic/intimate/quirky/gritty). Export `MOODS_MAX` (suggest 8) so tagging stays a signal rather than a checkbox sweep.
    - `EnergyLevel` type ('low' | 'medium' | 'high') + labels + values — deliberately mirroring `SoundIdentity.energy_level` so the two are directly comparable.
    - `VocalType` type ('vocal' | 'instrumental') + labels + values.
    - `TrackDescriptors = { moods: Mood[]; energy?: EnergyLevel | null; vocal?: VocalType | null; updated_at?: string }`.
    - Extend `TrackMetadata` with `descriptors?: TrackDescriptors` — purely additive; absent means untagged, exactly as `lyrics` and `recording` behave today.
    - `readDescriptors()` and `sanitizeDescriptors()` mirroring `readComposers()`/`sanitizeComposers()` in structure, defensiveness, and comment style.

    Vocal/instrumental must be an explicit artist-set field. Do NOT infer it from lyrics presence or the `zxx` language code — an instrumental version of a vocal track may still carry lyrics metadata, and absence of lyrics text does not mean the recording has no vocals.

    Write descriptors.test.ts to the behavior block first (RED), then implement.
  </action>
