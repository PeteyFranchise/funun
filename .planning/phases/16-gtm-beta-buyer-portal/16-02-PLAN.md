# 16-02: License Request Schema, Lifecycle & API

## Objective

Plan the first-class `license_requests` workflow so buyer demand becomes structured product data, not email threads or external-form residue.

## Scope

- Define buyer, artist, project, track, and opportunity references.
- Define usage fields: project type, usage description, media/placement, territory, term, exclusivity, budget range, need-by date, brief/reference upload, and notes.
- Define internal fields: stage, assigned owner, quote amount, deal notes, contract document link, closed reason, and timestamps.
- Define buyer-facing and admin-facing API routes.
- Define privacy rules for cross-user visibility.

## Acceptance Criteria

- License-request schema and lifecycle states are documented.
- API route family is specified for buyer create/read/update and admin review/update.
- RLS/API privacy doctrine is explicit.
- Manual/external-form fallback is allowed only if it calls the same API or writes the same tables.
- Contract Locker link fields are included from day one.

## Validation

- Threat model includes buyer sees another buyer's request, artist sees another artist's deal, and private project request bypass.
- Plan includes tests for blocked/private/non-public references.
