# ⚠️ SUPERSEDED BY D-18c (2026-07-20) — READ BEFORE ANY WORK

**This plan no longer builds a SignWell adapter. Sync licenses use the DocuSeal adapter that Phase 17 already builds.**

Everything below referencing SignWell — `lib/esign/providers/signwell.ts`, `SIGNWELL_API_KEY`, `SIGNWELL_WEBHOOK_SECRET`, the SignWell account setup, the webhook-scheme documentation read — is **void**. Do not create a SignWell account. Do not add those env vars.

## What this plan does instead

1. **Reuse `lib/esign/providers/docuseal.ts`** (built in 17-06, live-verified). No new adapter, no new credentials — `DOCUSEAL_API_KEY` and `DOCUSEAL_WEBHOOK_SECRET` are already configured.
2. **Reuse the webhook route** from 17-07. Sync-license completions arrive on the same verified endpoint; dispatch on the envelope, not on a provider guess. (17-07 was already required to be document-type-agnostic per the contract-library note — this is that requirement paying off.)
3. **Keep everything else in this plan unchanged**: the sync-license PDF template, `assertCounselReviewedForProduction()`, the Contract Locker destination (D-08), embedded + mobile-first signing (D-18b), and the mint route's `send_email: false` + `reply_to` amendment.
4. **Verify sequential signing before shipping.** Split sheets sign in parallel (`order: 'random'`, verified live). A sync license may need the buyer to countersign after the artist, or Funūn to sign last. DocuSeal documents `order: 'preserved'` for this but it is **NOT yet exercised** — test it against the live account before this plan's completion. A broken signing order on a revenue transaction is worse than on a split sheet.

## Why (D-18c rationale)

D-18b split providers on the premise that SignWell offered better legal defensibility for revenue transactions. **That premise was never verified** — SignWell was chosen before DocuSeal was evaluated, and no SignWell certificate was ever inspected. DocuSeal is verified in depth (live 3-signer async test; certificate carrying dual SHA256, per-signer IP/session/user-agent/timezone, email-verified flag, full timestamped event log, ESIGN/UETA/eIDAS positioning; API-completed signatures distinguished from hand-signed in the audit record). Maintaining two integrations to preserve an unverified asymmetry is the weaker position. DocuSeal is also cheaper at every volume ($0.20 vs $0.85/completion), already paid for, already white-labeled, and its mobile embed tested better (spike 006b found SignWell had first-paint horizontal overflow and small on-document tap targets).

**The `lib/esign/provider.ts` abstraction stays.** Consolidating is not lock-in — adding a second provider later is one adapter behind the existing interface. Accepted risk: vendor concentration.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DocuSeal -> webhook endpoint | Unauthenticated inbound HTTP that can assert a contract is signed. Same verified endpoint 17-07 hardened for split sheets. |
| browser -> embedded iframe | Signer interacts with vendor-hosted UI; the API key never reaches the browser. |
| admin -> signature initiation | Privileged action creating a legal instrument. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-38 | Spoofing | forged completion event fabricating a signed contract | critical | mitigate | Raw-body-first HMAC verification via the adapter's parseWebhook using the documentation-confirmed scheme with timing-safe comparison; no partial processing on failure. |
| T-16-39 | Repudiation | test-mode document treated as legally binding | high | mitigate | Test mode gated strictly on server environment, never client input; document records carry the mode; counsel review required before production claims. |
| T-16-40 | Elevation of Privilege | non-admin initiating a signature request | high | mitigate | verifyAdmin on the sign route plus stage and economics preconditions. |
| T-16-41 | Information Disclosure | DOCUSEAL_API_KEY exposure | high | mitigate | Server-only env var (never NEXT_PUBLIC_*); the browser only ever renders a returned per-signer embed URL and never calls the vendor directly. Unchanged from 17-06. |
| T-16-42 | Tampering | duplicate contract document from redelivered event | medium | mitigate | Idempotent handler keyed on existing linked document. |
| T-16-43 | Denial of Service | document insert failing on the type constraint | medium | mitigate | Explicit constraint-violation error branch naming the missing migration (RESEARCH Pitfall 7). |
| T-16-SC | Tampering | npm/pip/cargo installs | low | accept | No package installs — this plan writes no adapter at all (D-18c), and 17-06's DocuSeal adapter is a fetch-based in-repo module with the MIT @docuseal/react embed already vetted. |
</threat_model>
