<objective>
Implement the first live e-sign provider: a thin in-repo SignWell adapter behind the existing lib/esign/provider.ts abstraction (D-18a), the standard sync-license PDF Funūn drafts from (D-18), embedded signing so both parties sign in-app, and the completion webhook that files the signed contract into Contract Locker (D-08).

Purpose: The adversarial review flagged contract signing as the likely blocker for the first buyer deal, and the provider decision was stale in the docs. SignWell was chosen because it supports embedded signing pay-as-you-go while Dropbox Sign gates it behind a $300/mo tier. This plan also closes the documentation drift.
Output: The SignWell adapter with tests, the sync-license PDF renderer, the admin send-for-signature action, the verified webhook, and a rewritten integration doc.
</objective>

  <action>
    Extend the EsignState provider union in lib/esign/provider.ts to include the SignWell provider id, and update readEsignState's provider-narrowing branch so a persisted SignWell state round-trips correctly instead of silently defaulting to the legacy value. Change nothing else about the interface contract — the whole point of the abstraction is that callers are unaffected.

    Implement lib/esign/providers/signwell.ts as a plain fetch-based adapter implementing EsignProvider (createRequest, downloadSignedPdf, parseWebhook). Add no npm dependency: SignWell publishes no first-party Node SDK, and depending on an unofficial third-party wrapper for a legally load-bearing signing flow is worse than a small auditable in-repo module — this matches the design already described in the provider file's own header comment.

    createRequest posts the Funūn-generated PDF as the document payload with embedded signing enabled and draft disabled, using the authentication header name confirmed in the checkpoint. Send Funūn's own rendered PDF rather than a SignWell-hosted template so the contract stays version-controlled in git. Gate test mode strictly on the server environment (non-production), never on a client-supplied parameter — a staging signature must never be mistaken for a binding one (RESEARCH Pitfall 6). Return the request id and the per-signer embedded signing URLs the interface expects.

    parseWebhook verifies the signature using the confirmed scheme with Node's built-in crypto HMAC and a timing-safe comparison, throwing on an invalid signature per the interface contract. Do not reuse the svix helper — it is Resend-specific and would misparse SignWell's headers.

    signwell.test.ts mocks fetch to cover: createRequest sends embedded signing enabled; test mode is on outside production and off in production; parseWebhook throws on a tampered signature and returns the mapped event type on a valid one; an unknown provider event maps to the interface's catch-all type.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SignWell -> webhook endpoint | Unauthenticated inbound HTTP that can assert a contract is signed. |
| browser -> embedded iframe | Signer interacts with vendor-hosted UI; the API key never reaches the browser. |
| admin -> signature initiation | Privileged action creating a legal instrument. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-38 | Spoofing | forged completion event fabricating a signed contract | critical | mitigate | Raw-body-first HMAC verification via the adapter's parseWebhook using the documentation-confirmed scheme with timing-safe comparison; no partial processing on failure. |
| T-16-39 | Repudiation | test-mode document treated as legally binding | high | mitigate | Test mode gated strictly on server environment, never client input; document records carry the mode; counsel review required before production claims. |
| T-16-40 | Elevation of Privilege | non-admin initiating a signature request | high | mitigate | verifyAdmin on the sign route plus stage and economics preconditions. |
| T-16-41 | Information Disclosure | SignWell API key exposure | high | mitigate | Server-only env var; the browser only ever renders a returned embedded URL and never calls the vendor directly. |
| T-16-42 | Tampering | duplicate contract document from redelivered event | medium | mitigate | Idempotent handler keyed on existing linked document. |
| T-16-43 | Denial of Service | document insert failing on the type constraint | medium | mitigate | Explicit constraint-violation error branch naming the missing migration (RESEARCH Pitfall 7). |
| T-16-SC | Tampering | npm/pip/cargo installs | low | accept | No package installs — the adapter is deliberately built on fetch because no first-party SignWell SDK exists and unofficial wrappers were rejected (RESEARCH Package Legitimacy Audit). |
</threat_model>
