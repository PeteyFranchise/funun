<objective>
Build the money rails: Stripe Connect Express onboarding for artists (transfers-only), a buyer Checkout Session that pays Funūn with an automatic destination split, and the signature-verified webhook that records payment completion (D-17, D-17a, D-20).

Purpose: D-17 makes Funūn the merchant of record in the middle of every deal, and D-17a puts Connect in from day one rather than deferring to manual payouts. All fee math is pure and unit-testable so the economics are proven without live keys; only the live wiring needs credentials.
Output: The Connect module with tests, artist payouts onboarding, the admin-triggered buyer payment route, the Stripe webhook, and a migration recording connected accounts and payment state.
</objective>

  <behavior>
    buildExpressAccountParams(country):
    - requests the transfers capability.
    - does NOT request the card payments capability (the artist's connected account never charges anyone; Funūn is merchant of record — RESEARCH Pitfall 5).
    - sets the account type to express.

    buildDestinationChargeParams(deal, connectedAccountId):
    - the application fee equals the commission computed by computeNetFee from the deal's stored gross fee and commission percentage — never a value passed in by a caller.
    - the transfer destination is the artist's connected account id.
    - the charge amount equals the deal's stored gross fee.
    - throws a descriptive Error when the deal has no gross fee, no commission percentage, or no connected account id (a deal cannot be charged before it has economics and a payee).
    - application fee plus the amount transferred to the artist always reconstitutes the gross exactly (no rounding leak).
  </behavior>

Export the two pure parameter builders from the behavior block plus thin I/O wrappers: createExpressAccount, createAccountLink (onboarding type, with refresh and return URLs pointing at the payouts settings routes), and createDealCheckoutSession.

    Use classic Express accounts (account type express) — the stable, documented path the installed SDK targets. Do not adopt the newer configurable-accounts API surface mid-phase (RESEARCH Assumption A7: watch, do not adopt).

    Bill the buyer with a Checkout Session in payment mode carrying the destination transfer and application fee on the underlying PaymentIntent. This resolves RESEARCH Open Question 2 in favour of Checkout over Stripe Invoicing: lower build surface at three-to-five-deal beta volume, and Stripe hosts the payment page so no card data touches Funūn. Read the raw request body as text FIRST, verify it with the Stripe SDK's own constructEvent against STRIPE_WEBHOOK_SECRET, and only then parse and act — a body parsed and re-serialized before verification will not byte-match what Stripe signed (the ordering already proven correct in app/api/webhooks/resend/route.ts). <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Stripe -> webhook endpoint | Unauthenticated inbound HTTP that can mark money as received. |
| buyer browser -> Checkout | Payment happens on Stripe-hosted pages; no card data reaches Funūn. |
| admin -> charge creation | Privileged action creating a real money movement. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-32 | Spoofing | forged Stripe webhook marking a deal paid | critical | mitigate | Raw-body-first constructEvent verification with the signing secret; no partial processing on failure; idempotent redelivery handling. |
| T-16-33 | Tampering | client-influenced application fee zeroing commission | critical | mitigate | Fee recomputed server-side from the deal's stored commission percentage at charge-creation time; builders throw when economics are absent. |
| T-16-34 | Elevation of Privilege | non-admin triggering a charge | high | mitigate | verifyAdmin on the payment route plus precondition checks (signed contract, onboarded payee) before session creation. |
| T-16-35 | Information Disclosure | secret key or connected account id exposed to the browser | high | mitigate | All Stripe calls server-side only; connected account id ungranted to authenticated/anon in migration 065; no NEXT_PUBLIC exposure. |
| T-16-36 | Repudiation | fraudulent connected account receiving payouts | medium | transfer | KYC handled entirely by Stripe-hosted Account Links; Funūn collects no identity documents and relies on Stripe's platform vetting model. |
| T-16-37 | Tampering | duplicate payout from redelivered event | high | mitigate | Idempotent webhook handler keyed on existing payment state. |
| T-16-SC | Tampering | npm/pip/cargo installs | low | accept | No package installs — Stripe 17.7.0 is already a project dependency (RESEARCH Package Legitimacy Audit: approved, no change). |
</threat_model>
