  <name>Task 1: POST /api/buyer/requests — validated creation with pre-cleared-terms matching</name>
  <files>app/api/buyer/requests/route.ts</files>
  <action>
    Resolve the caller's buyer_members row; reject non-members. Authorize BOTH tiers to create requests — submitting is default verified-buyer reach for requester and approver alike (D-14a); only terms/budget approval and sign-off is approver-gated, and that happens later in the admin flow, not here.

    Validate the payload with a Zod schema (the project-wide validation standard) covering exactly the D-07 buyer-supplied dimensions: target vault_project_id, the track ids being requested, usage types, territories, term months, exclusivity, budget in cents, need-by date, and free-text buyer notes. Validate usage/territory members against the vocabularies in lib/deals/schema.ts so they are comparable with pre-cleared terms. Explicitly REJECT any client-supplied stage, owner_id, commission_pct, gross_fee_cents, artist_net_cents, or matched_precleared — these are server-owned authority fields (migration 063 revoked client UPDATE; the route must not reintroduce the hole at the application layer).

    Authorize the target: re-run the same rights-ready and Phase 13 visibility/block checks the catalog route applies before accepting a request, so a buyer cannot request a license on a private, unready, non-owned, or blocking artist's project by posting an id they guessed (the 16-VALIDATION abuse cases V16-03). Verify every requested track id actually belongs to the target project — one request equals one project, many tracks.

    Load the project's pre-cleared terms and call matchesPreclearedTerms(). Persist the resulting boolean into matched_precleared and set the initial stage: a matching request enters the pipeline flagged as pre-cleared, a non-matching request (including the no-terms-set case) is flagged for admin negotiation first (D-15a — Funūn admins play the agency role and align terms with the buyer before the artist is engaged). Store the mismatch reasons so the admin queue in plan 16-07 can show exactly which dimensions are out of band. The server, not the client, decides this.

    Insert the license_requests row and its license_request_tracks children with the service-role client after all checks pass. Then, as a best-effort side effect wrapped in try/catch AFTER the primary insert (the lib/social/activity-emit.ts convention), emit the artist notification via createNotification using buildLicenseRequestNotification — a notification failure must never fail the request creation.
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| buyer session → request creation | Untrusted commercial payload creating first-class deal data. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-22 | Tampering | client-supplied stage/commission/matched flag on create | critical | mitigate | Zod schema accepts only buyer dimensions; authority fields rejected at the route and revoked at the DB (migration 063). Match verdict computed server-side. |
| T-16-23 | Information Disclosure | request against a private/blocked/non-ready project | high | mitigate | Target re-authorized through the same rights-ready + visibility + block path as catalog before insert; track ids verified to belong to the project. |
| T-16-24 | Information Disclosure | cross-org request read | high | mitigate | Request detail scoped to the caller's org with a 404 (not 403) response for anything else. |
| T-16-25 | Information Disclosure | artist's pre-cleared minimum fee leaked to buyer | medium | mitigate | Composer never renders terms or the match verdict; match result is admin-facing only. |
| T-16-26 | Denial of Service | request spam against an artist | medium | accept | Beta is invite-only with admin-created orgs (D-12) and founder-mediated deals; volume-based rate limiting deferred until deal volume justifies it. |
| T-16-SC | Tampering | npm/pip/cargo installs | low | accept | No package installs in this plan. |
</threat_model>
