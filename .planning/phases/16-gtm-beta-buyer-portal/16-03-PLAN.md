# 16-03: Buyer Portal MVP & Safe Request Entry Points

## Objective

Plan the buyer-facing portal MVP: onboarding, request creation, request tracking, and safe "Request License" entry points on public/discoverable surfaces.

## Scope

- Decide portal route naming, likely `/buyer`, `/sync`, or `/license`.
- Define buyer dashboard: request list, status, missing-info prompts, and next steps.
- Define request composer UX.
- Define "Request License" entry points on public profile, release/project cards, Green Room feed cards, and buyer search/discovery if allowed.
- Define what buyer can search before verification versus after verification.

## Acceptance Criteria

- Buyer can submit a request without emailing the founder.
- Buyer can track request status.
- Request entry points reuse the same request composer/API.
- Buyer cannot request private, blocked, hidden, or unlicensable items.
- UI copy distinguishes request submission from guaranteed clearance.

## Validation

- Browser UAT with a test buyer account.
- API adversarial tests for request entry points.
- Accessibility and mobile smoke test for portal MVP.
