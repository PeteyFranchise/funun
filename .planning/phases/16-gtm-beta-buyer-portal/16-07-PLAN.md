  <name>Task 1: Deal-stage state machine</name>
  <files>lib/deals/stage-machine.ts, lib/deals/stage-machine.test.ts</files>
  <behavior>
    isLegalTransition(from, to):
    - forward moves along the D-16a pipeline are legal (submitted to in_negotiation to terms_agreed to contract to closed_won).
    - any non-terminal stage may transition to the declined/closed_lost terminal stage.
    - a terminal stage (closed_won or closed_lost) transitions to nothing — every outbound move is illegal.
    - skipping a stage forward (for example submitted directly to contract) is illegal.
    - a same-stage self-transition is illegal (a no-op write must not pass the gate).
    - an unknown stage value on either side is illegal (fail closed).

    requiredFieldsForStage(stage):
    - the contract stage requires a gross fee and a commission percentage to be present (a deal cannot reach paper without economics).
    - the closed_won stage requires a linked contract document.
  </behavior>
  <action>
    Implement the stage machine as pure functions in lib/deals/stage-machine.ts over the DEAL_STAGE_VALUES tuple from lib/deals/schema.ts — no Supabase client, no I/O. Export isLegalTransition, requiredFieldsForStage, and a helper returning the legal next stages for a given stage so the UI renders only valid actions from the same source of truth the server enforces.

    This guard is what makes the deal pipeline real rather than a free-text status column: the API route in Task 2 must call it, and the UI must derive its buttons from it, so a stale open tab or a crafted request cannot jump a deal straight to closed. Write the tests to the behavior block first (RED), then implement to green.
