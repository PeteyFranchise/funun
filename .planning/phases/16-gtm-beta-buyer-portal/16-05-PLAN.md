<objective>
Build the buyer's discovery surface: a filtered, rights-ready catalog browse (D-16) and org-shared shortlists (D-14c), both reachable by either permission tier (D-14a).

Purpose: This is the demand side's entry point and the phase's most privacy-sensitive read surface — Phase 13's trust/safety gates (D-09, shipped) must hold here exactly as they do on artist-facing surfaces. Discovery stays deliberately beta-safe: filtered browse only, no free-text search ranking, no messaging, no non-public availability signals (D-05/D-14a).
Output: The isRightsReady helper with tests, the catalog and shortlist APIs, and the two portal pages.
</objective>

  <files>lib/deals/catalog.ts, lib/deals/catalog.test.ts</files>
  <behavior>
    isRightsReady(project, stage3Result):
    - project not public → false, regardless of readiness.
    - public but readiness score below the threshold → false.
    - public, readiness at or above the threshold, but stage3.canContinue false (e.g. an uncleared flagged sample blocking distribution) → false.
    - public, readiness at or above the threshold, stage3.canContinue true → true.
    - readiness exactly at the threshold boundary → true (inclusive comparison).
    - missing/null readiness score → false (fail closed).
  </behavior>
  <action>
    Implement isRightsReady in lib/deals/catalog.ts as the SINGLE named helper expressing the rights-ready definition, exporting the threshold as a named constant alongside it. Document in a header comment that the threshold is deliberately TUNABLE in this one place: the beta definition is public AND readiness at or above the threshold AND computeStage3().canContinue, and product may raise or lower it after observing how much catalog surfaces (this resolves RESEARCH Open Question 3 / Assumption A4 as a decided default, not a guess). Do NOT introduce a separate boolean flag column on vault_projects — a parallel flag desyncs from the readiness pipeline the first time either changes (RESEARCH Don't Hand-Roll).

    The helper is pure: it accepts an already-fetched project shape and an already-computed Stage3Result, so callers do the I/O and this stays unit-testable. Reuse computeStage3 from lib/vault/stage3.ts at the call site rather than re-deriving clearance logic.

    Also export the catalog filter vocabulary and a buildCatalogFilter helper that turns validated filter params (genre, mood/energy, vocals, usage cleared) into a normalized filter object the route applies. Explicitly provide no free-text query parameter — filtered browse only (D-16); admin-curated collections may layer on later but are not built here.
