<objective>
Build the inbound-signal side of the phase (PITCH-06): the first webhook route in this codebase (`/api/webhooks/resend`, svix-signature-verified, flips `email_valid=false` on hard bounce), and the token-authenticated one-click response links from the pitch email footer — accept (bare click), decline (optional-reason page), and unsubscribe (sets `do_not_pitch`). Accept/decline notify the artist via the existing in-app + email notification pattern (D-14).

Purpose: These are the curator-driven state transitions that make pitch history meaningful. The webhook is the single genuinely-new security surface in the phase — it MUST read the raw body before parsing and verify the svix signature before any DB write (RESEARCH Pitfall 2 + the forged-webhook threat), and it degrades gracefully (503) until RESEND_WEBHOOK_SECRET is configured (D-22/Pitfall 4). This plan READS response tokens (generated by 06-04's send route) and resolves them one-time; it does not generate tokens.
Output: A verified bounce-webhook path and three public token-response routes/pages, all reusing the existing notification and token patterns.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public internet → /api/webhooks/resend | Anyone can POST here; only a valid svix signature distinguishes Resend from an attacker |
| public token link → pitch_history / curators mutation | Unauthenticated token holders drive status transitions; the token is the sole authorization |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-16 | Spoofing / Tampering | forged POST to `/api/webhooks/resend` | high | mitigate | Mandatory svix signature verification against `RESEND_WEBHOOK_SECRET` (constant-time HMAC) BEFORE any DB write; 400 on any verification failure, 503 when unconfigured — never a best-effort accept (RESEARCH Pitfall 2 + Security Domain) |
| T-06-03 | Tampering / Repudiation | accept/decline response-token replay | medium | mitigate | 256-bit `response_token` (generated in 06-04) + one-time-use guard: `status !== 'pending'` → 410; brute force infeasible at 256 bits |
| T-06-17 | Tampering | unsubscribe-link abuse | low | accept | Setting do_not_pitch is non-destructive (curator stays in the directory) and idempotent; the token maps to one specific curator via the pitch, so worst case is a self-unsubscribe of an already-pitched curator — accepted as low impact (D-20) |
| T-06-18 | Information Disclosure | webhook error / event body surfacing | low | mitigate | The route returns only `{ ok }` / minimal status codes; the raw Resend event body is never echoed to the caller |
</threat_model>
