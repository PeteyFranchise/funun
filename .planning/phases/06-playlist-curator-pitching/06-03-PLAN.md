<objective>
Build the artist-facing curator directory: the `/curators` page (server component, genre + platform filtering via searchParams — PITCH-01), the authenticated-artist directory GET API that returns directory-safe curator fields joined with each curator's 90-day response rate (PITCH-04 display side), the `CuratorCard` display component (reused later by the composer's multi-select), and the `CuratorDirectory` client wrapper that renders the filter bar and grid.

Purpose: This is the browse surface artists use to discover curators before pitching. D-07 keeps the composer OUT of `/curators` — this page is browse-only (no composer modal, no selection mode here). The response-rate number and genre focus are the two data points an artist uses to judge fit, so both must be correct and correctly hidden when there is no data.
Output: A working, filterable read-only curator directory with correct response-rate and reach-signal display states.
</objective>

lib/curators/response-rate.ts: export `computeResponseRates(service, curatorIds?: string[]): Promise<Map<string, number>>` returning curatorId → integer response-rate percent, using the D-13 formula: over pitch_history rows where sent_at is within the last 90 days, rate = round( count(status IN ('accepted','declined')) / count(*) * 100 ). Implement by selecting `curator_id, status, sent_at` from pitch_history filtered to `sent_at >= <ISO of now minus 90 days>` (compute the cutoff in JS: `new Date(Date.now() - 90*24*60*60*1000).toISOString()`), then aggregate in JS grouped by curator_id (the project has no test framework and favors app-code aggregation — see lib/vault/readiness.ts style). app/api/curators/route.ts: GET handler. Use createApiClient(); getUser() → 401 if no user (this is authenticated-artist-only, NOT public). Read optional `genre` and `platform` repeatable searchParams from the request URL. Query curators via the user-scoped client (RLS SELECT USING(true) permits it) with an EXPLICIT column projection — select only directory-safe fields: `id, name, playlist_name, playlist_url, platform, genre_focus, reach_signal, reach_fetched_at, drift_flagged, do_not_pitch, email_valid, claimed_by`. Apply platform filter with `.in('platform', platforms)` only when platforms is non-empty and every value is in PLATFORM_VALUES (reject unknown platform values). Apply genre filter with `.overlaps('genre_focus', genres)` when genres non-empty. Then call `computeResponseRates(createServiceClient())` and attach `response_rate` (Map value, or null when absent) to each curator. Return `{ data }` where data is `CuratorWithRate[]`. Note: claimed_by is projected as a boolean-ish presence signal — map it to `claimed: boolean` (claimed_by !== null) in the response rather than leaking the raw user id; do not return the raw claimed_by UUID.</action>
