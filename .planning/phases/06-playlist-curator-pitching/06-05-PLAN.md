<objective>
Build the curator claim flow and the curator self-serve portal (PITCH-05): claim-token issuance (admin issues an invite → 72h-expiry token on the curator row), the public `/curators/claim/[token]` page + verify/create API (creating a lightweight magic-link Supabase account with `app_metadata.role='curator'` set at creation so 06-01's `handle_new_user()` branch keeps curators out of `artist_profiles`), the curator self-serve PATCH API (narrow `CURATOR_SELF_EDITABLE_FIELDS` allowlist + `claimed_by` ownership scoping), and the `(curator-portal)` route group with its own auth-checking layout, a profile-edit form, and a read-only "pitches I've received" view.

Purpose: Claiming turns a static directory record into a self-maintained profile (D-18/D-19) and is the curator-onboarding growth loop. The single highest-risk correctness point is that curator account creation must NOT create an artist profile — which depends entirely on setting `app_metadata.role` at `createUser` time (RESEARCH Pitfall 1) and on the curator-portal being gated by its own layout, never by the artist `middleware.ts` prefix list (RESEARCH Pitfall 3).
Output: A working claim → magic-link account → self-serve portal loop, with mass-assignment and IDOR protection on the self-edit path.
</objective>

| curator self-serve PATCH → curators table | A curator must edit ONLY their own row and ONLY allowlisted fields |
| curator-portal routes → artist auth model | Curator routes are outside middleware.ts's artist gate; the portal layout is the sole authority |
