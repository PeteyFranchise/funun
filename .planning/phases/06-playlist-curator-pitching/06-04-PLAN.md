<objective>
Build the pitch composer and send pipeline that lives inside `/launchpad/[projectId]` (D-07 — never on `/curators`): the shared token generator (`lib/curators/tokens.ts`, owns both claim and response token shapes), the AI 150-word pitch-note drafter (reusing the PitchPlug JSON-fence + demo-mode pattern), the send route (`/api/pitches`) that does ownership + 3-gate server-side re-validation + duplicate-send guard + `pitch_history` row creation with a per-row `response_token` + email delivery, the `lib/email` `from`-override, and the `PitchComposer` + `PitchHistoryList` UI wired into the launchpad room (PITCH-02, PITCH-03).

Purpose: This is the core artist action of the phase. The composer must make the locked 150-word playlist-specific note gate meaningful (a real AI-drafted note, not boilerplate) and enforce it on the server, not just the client. This plan OWNS `pitch_history` row creation, so it also owns generating `response_token` (consumed by 06-06's accept/decline links) — 06-06 does not generate tokens, it only reads and resolves them.
Output: A working end-to-end pitch composer + send + per-project history, degrading gracefully when PITCH_FROM_EMAIL is unset (D-22).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| composer client → /api/pitches | The client's Send-button disabled state is untrusted; the note length + selection must be re-validated server-side |
| artist → another artist's project/track | Send must only pitch tracks the caller owns |
| artist → curator directory (send target) | do_not_pitch / bounced / duplicate curators must be blocked at the send boundary |
