# Phase 6 — UI Design Contract: Playlist Curator Pitching

> Visual and interaction contract for Phase 6 frontend work. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — hand-rolled Tailwind components, same system used in Phases 1–5 |
| Icon library | inline SVG (hand-rolled, see `components/nav/icons.tsx`) |
| Font | Inter via `next/font/google`, variable `--font-sans` |

No shadcn initialization. `components.json` was checked and is absent (confirmed by direct file scan this session). The project uses a bespoke dark design system defined in `tailwind.config.ts` and `app/globals.css`, already carried through Wave 2 and Phase 5 of Wave 3. This phase does not introduce a component library — it extends the existing hand-rolled system. This decision is inherited from Phase 5's explicit "Tool: none" declaration and five phases of consistent precedent; re-litigating it per phase would fragment the visual language across the shared `/launchpad/[projectId]` route this phase lives inside.

**New surface type this phase introduces:** unauthenticated, session-less public pages (`/pitch/accept/[token]`, `/pitch/decline/[token]`, `/curators/claim/[token]`) and a curator-portal account type. These reuse the existing `app/(auth)/layout.tsx` minimal centered-card shell (`flex min-h-screen items-center justify-center px-6 py-12`, `max-w-sm`, Funūn wordmark header) rather than inventing a new public layout.

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline chip padding, genre-tag internal padding |
| sm | 8px | Compact element spacing, badge row gaps, card grid gap (`gap-3`) |
| md | 16px | Default element spacing, curator card body padding, form field stacks |
| lg | 24px | Section padding, composer panel internal spacing |
| xl | 32px | Layout gaps between directory sections (filters vs. results) |
| 2xl | 48px | Major section breaks (e.g. between pitch history and directory on the launchpad pitch tab) |
| 3xl | 64px | Page-level spacing |

### Approved Spacing Exceptions

| Value | Location | Business Justification |
|-------|----------|------------------------|
| 9px (`px-9` = 36px) | Topbar horizontal padding on `/curators` and `/admin/curators` | Project-standard topbar padding established in `Topbar.tsx`. New pages reuse the shared `Topbar` component unchanged — inheriting its exceptions is required to avoid a one-off mismatched header across the app. |
| 22px (`pb-[22px]`) / 26px (`pt-[26px]`) | Topbar vertical padding, same pages | Same justification — inherited from the reused `Topbar` component, not a new decision. |
| 18px (`rounded-[18px]`) | Curator directory card radius | Matches `CollaboratorCard.tsx`'s established `rounded-[18px]` treatment for roster-style cards. Curator cards are visually the same "person/entity record" card type and must match, not introduce a third card-radius value alongside `rounded-xl` (DocumentCard/PitchCard) and `rounded-[14px]` (Phase 5 checklist rows). |

These exceptions are **locked** — do not "fix" them to the nearest multiple of 4 during implementation. All are inherited from existing reused components, not new deviations introduced by this phase.

Source: `components/layout/Topbar.tsx`, `components/collaborators/CollaboratorCard.tsx` — read directly.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-[14px]` / `text-sm`) | 400 (regular) | 1.5 |
| Label | 12px (`text-[12px]`) | 700 (bold), uppercase, `tracking-[.14em]` | 1.3 |
| Heading (card title / curator name / section header) | 14px (`text-[14px]`) | 700 (bold) | 1.3 |
| Display (page title, via reused `Topbar`) | 27px (`text-[27px]`) | 800 (extrabold — inherited, see note) | 1.1 |

Permitted weights for **new** Phase 6 components: **400 (regular)** and **700 (bold)** only — matches the tightened convention already established for `/launchpad/[projectId]` (Phase 5) and `/admin/checklist` (`ChecklistAdmin.tsx` uses `font-bold` exclusively). Since the pitch composer lives inside `/launchpad/[projectId]` and `/admin/curators` sits directly beside `/admin/checklist`, using a different weight system would create visible inconsistency within the same route tree.

**Note on Display role:** the shared `Topbar` component (reused verbatim, not modified) hard-codes `font-extrabold` (800) for its `<h1>`. This is inherited, not a new weight introduced by this phase — do not add a third weight value anywhere else.

Notes:
- Curator card name: `text-[14px] font-bold text-white` — mirrors `CollaboratorCard.tsx`'s name treatment exactly
- Curator card meta (genre focus, reach signal): `text-[12.5px] text-lavdim` — mirrors `CollaboratorCard.tsx`'s PRO-label line
- Section/filter labels ("Genre", "Platform"): `text-[12px] font-bold uppercase tracking-[.14em] text-lavdim` — matches Phase 5's section-header pattern
- Word counter in composer: `text-[13px] text-lavdim` normally; `text-[13px] font-bold text-rose-300` when over 150 words
- Public page (claim/accept/decline) body copy: `text-sm text-white/70` — matches `(auth)/layout.tsx` subtitle convention
- Badge text (status pills): `text-xs font-bold` inside pill — matches `DocumentCard.tsx` `STATUS_META` badge pattern

Source: extracted from `CollaboratorCard.tsx`, `DocumentCard.tsx`, `Topbar.tsx`, `ChecklistAdmin.tsx`, `app/(auth)/layout.tsx`.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0a0a0f` (`ink`) | Page background across `/curators`, `/admin/curators`, composer panel, public claim/accept/decline pages |
| Secondary (30%) | `#0E0D1E` (`card`) / `#1A1838` (`card2`) | Curator directory cards, composer panel surface, admin curator rows, filter bar surface |
| Accent (10%) | `#818CF8` → `#D946EF` (`grad`) | Specific elements listed below — nothing else |
| Destructive | `#F43F5E` (rose-500) | Admin "Delete curator" confirmation only; also reused (not accent) for the "email invalid" / "declined" semantic badges below |

Accent reserved for:
1. **"Send pitch" primary CTA button** in the composer — the single most important action of this phase, `bg-grad`
2. **Admin "Add curator" CTA button** on `/admin/curators` — mirrors Phase 5's "Add checklist item" gradient button exactly
3. **"Claim your profile" CTA button** on the public `/curators/claim/[token]` page — this is the curator-onboarding growth loop referenced in `PROJECT.md` and deserves the same visual weight as the composer Send button
4. **Response-rate percentage number** on curator cards (e.g. "62%") — uses the existing `.gtext` gradient-text utility class, same treatment as Phase 5's completion-fraction text

Do **not** use accent for: filter chips, genre tags, curator card borders, "select curator" checkboxes, or the AI-draft "Regenerate note" button (that one is a secondary/ghost action, not primary).

Secondary semantic colors (from project palette, reused exactly — not accent):
- Pitch status `pending`: `border-amber-400/30 bg-amber-400/10 text-amber-300` — matches `DocumentCard.tsx` `STATUS_META.pending` verbatim
- Pitch status `accepted`: `border-emerald-400/30 bg-emerald-400/10 text-emerald-300` — matches `STATUS_META.signed` verbatim
- Pitch status `declined`: `border-rose-400/30 bg-rose-400/10 text-rose-300` — matches `STATUS_META.missing` verbatim
- Curator email invalid (hard bounce): `border-rose-400/30 bg-rose-400/10 text-rose-300` pill, "Email bounced"
- Genre drift badge: `border-amber-400/30 bg-amber-400/10 text-amber-300` — matches the codebase's existing amber = "needs attention" convention (same as `CollaboratorCard`'s missing-IPI left-border and `DocumentCard`'s pending badge)
- "Funūn member" / claimed curator profile badge: `border-brandindigo/30 bg-brandindigo/10 text-brandindigo` pill — reuses `CollaboratorCard.tsx`'s "Funūn member" badge verbatim, labeled "Claimed profile"
- Admin "flagged inactive": `border-rose-400/30 bg-rose-400/10 text-rose-300` pill, "Inactive"
- Unsubscribed / do-not-pitch: `border-white/15 bg-white/5 text-white/50` neutral pill, "Unsubscribed" — deliberately not a semantic-red error color since this is a curator choice, not a fault
- "Already pitched" disabled option label: `text-white/30` — muted, non-interactive

## Component Interaction Contracts

### CuratorCard (`/curators` directory grid)

- Card shape: `rounded-[18px] border border-hair bg-card p-4` — same shell as `CollaboratorCard.tsx`
- Grid: `grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 gap-3`
- Content order top to bottom: name (bold, 14px) → playlist/channel name + platform icon → genre focus chips (max 3 visible, "+N more" overflow) → reach signal line (muted, approximate — see States) → response-rate badge (bottom-right, `.gtext` if ≥1 pitch on record, hidden if zero)
- Genre focus chips: `rounded-full border border-hair bg-card2 px-2 py-0.5 text-[11px] text-lav`
- Drift-flagged curators show a small amber warning icon + tooltip text next to the name (advisory only — does not block card interaction)
- Card has no click-through detail page in v1 — selection happens via checkbox/toggle when the multi-select mode is active (composer entry point), otherwise the card is browse-only (no click target)
- "Already pitched" curators (when arriving from a project-scoped context): card gets `opacity-60`, checkbox `disabled`, and a `text-white/30` label appended below the name: "Already pitched · {status label}" (D-08, locked)

### Filter bar (`/curators`)

- Horizontal pill row above the grid: `flex flex-wrap gap-2` — genre multi-select chips + platform multi-select chips
- Platform enum (finalized this session, Claude's Discretion item resolved): `spotify` ("Spotify"), `apple_music` ("Apple Music"), `youtube_music` ("YouTube Music"), `soundcloud` ("SoundCloud"), `blog_other` ("Blog / Independent")
- Active filter chip: `border-brandindigo/40 bg-brandindigo/10 text-brandindigo` (accent used here only because it is the "currently selected" state indicator, same role as active-nav — not a competing accent use)
- Inactive filter chip: `border-hair bg-card2 text-lav hover:border-hairstrong`
- Filters update the grid via re-fetch (server component + searchParams, same pattern as `/launchpad` global page), not client-side array filtering

### Pitch Composer (inside `/launchpad/[projectId]`)

- Entry point: a "Pitch to curators" action/tab within the project room (D-06/D-07 — composer always lives here, never on `/curators`)
- Track selector: dropdown/select listing every track in the project (D-09 — no lead-track restriction)
- Curator multi-select: reuses the CuratorCard grid in a compact list mode with checkboxes; already-pitched curators for the selected track appear disabled + labeled per D-08
- AI-draft note: "Draft a pitch note" ghost button (`border border-white/15 px-3 py-1.5 text-xs text-white/70`, matches `PitchCard.tsx`'s secondary-button style) triggers the Claude-drafted 150-word note into an editable `textarea`
- Note textarea: `rounded-lg border border-white/15 bg-white/[0.03] px-3 py-2 text-sm text-white` — matches `PitchCard.tsx` input styling exactly; `resize-none`, auto-grows
- Live word counter pinned below the textarea, right-aligned: "{count}/150 words" — `text-lavdim` under 150, `font-bold text-rose-300` at/over 150
- **Send button gating (locked, PITCH-02):** disabled unless (a) at least one curator selected, (b) note is non-empty, (c) word count ≤ 150. Server re-validates all three (never trust client state alone — see Security Domain in RESEARCH.md)
- Send button: `bg-grad px-5 py-3 text-[15px] font-bold text-white shadow-cta` (accent — see Color)
- Drift-flagged curator selected: inline advisory banner appears above Send, not blocking: `border border-amber-400/30 bg-amber-400/10 rounded-lg px-3 py-2 text-xs text-amber-300` — "This curator's genre focus may have shifted — double check fit." (D-17)
- On send: success toast/inline confirmation "Pitch sent to {N} curator(s)"; composer clears note + selection, curator cards flip to "Already pitched · Pending"

### Pitch History (within `/launchpad/[projectId]`)

- List/table of past pitches for the project: curator name, track, sent date, status badge (pending/accepted/declined per the three semantic colors above — no "opened" state, per D-10)
- Row style: `rounded-xl border border-white/10 p-4` — matches `DocumentCard.tsx` row shell
- Declined rows with a reason on file show the reason as muted secondary text below the row: `text-xs text-white/40 italic`

### Admin CuratorAdmin (`/admin/curators`)

- Mirrors `ChecklistAdmin.tsx` exactly in shell and interaction shape, minus drag-and-drop (curators have no manual ordering) — reuses only the inline add/edit/delete pattern, not the `@dnd-kit` sortable rows
- Each row: `rounded-xl border border-white/10 p-4` layout with name, email, platform, genre focus tags, reach signal, and a row of status pills (claimed/unclaimed, email valid/bounced, drift flag, flagged-inactive)
- Add/Edit form: inline expand below the list (not a modal) — same inline-expand pattern as `ChecklistAdmin.tsx`
- Delete: inline confirmation below the row (no modal) — "Delete this curator? This removes them from the directory permanently and does not affect existing pitch history. [Cancel] [Delete curator]"
- "Flag inactive" toggle: manual button with an advisory hint shown alongside it when a suggested signal exists (D-21): `text-[11px] text-amber-300` — "Suggested: no response in last 5 pitches" or "Suggested: email bounced" — the hint never auto-toggles the flag
- Genre drift badge: same amber pill as the artist-facing card, plus a "Reset baseline" ghost action for admin to acknowledge and re-baseline after review
- Claimed-profile rows get the same `brandindigo` "Claimed profile" badge as `CollaboratorCard.tsx`'s "Funūn member" badge — visually distinguishes claimed vs. unclaimed at a glance (locked requirement, PITCH-07)

### Public token pages: Claim, Accept, Decline

All three reuse `app/(auth)/layout.tsx`'s centered `max-w-sm` shell — no sidebar, no Topbar, no app chrome. This is a deliberate visual signal that these are outside the authenticated app.

- **Claim page** (`/curators/claim/[token]`): headline "Claim your curator profile", body copy explaining the benefit (self-serve edit + pitch visibility), single `bg-grad` CTA "Claim your profile" (accent). Expired/used token shows an inline error state instead of the CTA (no destructive styling needed — just muted copy, this isn't the curator's fault)
- **Accept page** (`/pitch/accept/[token]`): bare-click confirmation, no form (D-11). On load, action fires immediately server-side or via a single confirm button; success state shows "You've accepted this pitch — the artist has been notified." Already-responded token shows "This pitch was already responded to."
- **Decline page** (`/pitch/decline/[token]`): optional-reason `textarea` (skippable) + two actions: "Decline" (primary, `bg-white text-black font-bold` — not accent, this is a neutral/negative action, not a growth-loop CTA) and no visible "Skip" button separate from Decline — leaving the textarea empty and clicking Decline IS the skip (D-12)

### Curator Portal (self-serve, claimed accounts only)

- New route group, own layout, own auth check (`app_metadata.role === 'curator'`) — does not reuse `ArtistNav` or the artist `Topbar`
- Simple two-section layout: "Your profile" (editable form: genre focus, platform, playlist/channel URL, submission notes/bio — same input styling as the composer's note textarea) + "Pitches you've received" (read-only list, same row shell as artist-facing Pitch History but without a Send/action column)
- Editable-field allowlist enforced visually too — no fields for `email_valid`, `flagged_inactive`, or `reach_signal` appear in this form at all (not just disabled — absent, per the mass-assignment threat mitigation in RESEARCH.md)

## Accessibility Contracts

- Filter chips: `role="checkbox"` `aria-checked` on multi-select platform/genre pills, or native `<button aria-pressed>` — pick one pattern and use consistently across genre and platform filters
- Curator card checkbox (composer selection mode): min 44×44px touch target, `aria-label="Select {curator name} for pitch"` / `aria-label="{curator name} — already pitched, cannot re-select"` when disabled
- Composer word counter: `aria-live="polite"` region so screen-reader users hear the count update as they type
- Send button: `aria-disabled` reflects the same three-gate logic as the visual disabled state — never disable visually without `aria-disabled`
- Accept page bare-click action: still requires a focusable, keyboard-activatable `<button>` — "bare click" means minimal friction, not mouse-only
- Decline reason textarea: `aria-label="Reason for declining (optional)"`
- Admin delete confirmation: `role="alert"` on the inline expand, matching Phase 5's `ChecklistAdmin` convention
- Admin flag-inactive suggested hint: `aria-describedby` linking the hint text to the toggle button, so screen readers announce the suggestion before the action
- Drift warning badge/tooltip: not conveyed by color alone — always paired with the amber warning icon AND text, never a bare colored dot
- Curator portal profile form: standard form labeling (`<label htmlFor>`), no ARIA overrides needed beyond what native form elements already provide
