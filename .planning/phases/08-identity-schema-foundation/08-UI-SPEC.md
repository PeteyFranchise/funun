# Phase 8 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note (read first)

**Phase 8 is a schema/migration foundation phase with no user-facing requirement mapped to it** (per `ROADMAP.md` and `08-CONTEXT.md`). Every requirement in Phases 9–13 depends on this schema; success is verified structurally (migrations, RLS, column privileges), not against a UI.

**The only UI surface in this phase is `/admin/members`** (D-02): a minimal internal admin route for inviting industry members (email + display name + initial role badge picker), gated by the same `is_admin` pattern as `/admin/curators` and `/admin/checklist`.

**Everything else in this phase has zero visual footprint:**
- All `artist_profiles` column additions, `connections`/`blocks`/`notifications`/`dm_thread_reads`/`reserved_handles` tables, RLS policies, `no_block()` helper, column-privilege REVOKE/GRANT, and the `handle_new_user()` industry branch are pure database work — no rendered surface.
- The D-19 companion code fixes (`app/(artist)/settings/page.tsx`, `app/profile/page.tsx`, `app/u/[handle]/page.tsx`, `app/api/profile/route.ts`) are query/column-selection changes to prevent breakage from the column-privacy migration. They preserve the exact existing rendered output of those pages — no new visual elements, no layout change, no new copy. This contract does not cover them; the executor should treat them as a pure data-layer fix and visually diff against current behavior (before ≠ after is a bug).

This contract governs `/admin/members` only.

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — hand-rolled Tailwind components (project-wide convention; no shadcn anywhere in this codebase across 7 prior shipped phases) |
| Icon library | inline SVG (hand-drawn paths) — matches `ChecklistAdmin.tsx`'s grip-icon/arrow-icon convention; no icon package imported in existing `/admin` components |
| Font | `var(--font-sans)`, system-ui, sans-serif (`tailwind.config.ts`) |

**Why no shadcn:** `components.json` does not exist and no `/admin` surface in this codebase (`/admin/curators`, `/admin/checklist`, `/admin/tips`) uses shadcn — all are hand-rolled Tailwind against the project's custom design tokens (`ink`/`card`/`card2`/`lav`/`lavdim`/`brandindigo`/`brandfuchsia`/`hair`). Introducing shadcn for one small admin form would fork the visual system mid-milestone against `08-CONTEXT.md`'s explicit instruction to "follow existing `/admin/curators`/`/admin/checklist` visual conventions." Decision: continue the established hand-rolled pattern.

## Spacing Scale

Declared values (must be multiples of 4), sourced from the existing `/admin` surface:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, chip internal spacing |
| sm | 8px | `gap-2` between buttons/chips, `mb-3`/`mt-2` micro-spacing |
| md | 16px | `p-4` card padding, `gap-3` form field stacks, `mt-4` action-row spacing |
| lg | 24px | `mt-6` section top margin, page-level list gaps |
| xl | 32px | — (not used on this route; reserved for larger layouts) |
| 2xl | 48px | — (not used on this route) |
| 3xl | 64px | — (not used on this route) |

Exceptions:
- `py-1.5` (6px vertical) on compact buttons and role-picker chips — established project-wide for secondary/tertiary action density (seen in `CuratorAdmin.tsx`, `ProfileForm.tsx`); keep for consistency rather than rounding to 8px, since it matches every other admin/profile chip in the codebase.
- Page container uses `px-9 py-[30px]` (36px / 30px) — matches `/admin/curators`'s exact page wrapper; reuse verbatim, do not redesign.

## Typography

Sourced directly from `CuratorAdmin.tsx`/`ChecklistAdmin.tsx` — do not introduce new sizes.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Page heading (h1) | 24px (`text-2xl`) | 700 bold | 1.2 |
| Body / form input text | 14px (`text-[14px]`) | 400 regular | 1.5 |
| Label / button / section title | 13px (`text-[13px]`) | 700 bold | 1.4 |
| Meta / secondary text (email, dates, hints) | 12px (`text-[12px]`) | 400 regular | 1.4 |
| Micro (role-chip labels, status pills) | 11px (`text-xs`/`text-[11px]`) | 700 bold (chips/pills) or 400 (plain micro-copy) | 1.3 |

Weights (exactly 2, per contract): **400 regular** (body copy, meta text, placeholders) and **700 bold** (headings, labels, buttons, badges, emphasis). No `font-semibold`/`font-medium` on this route — matches `CuratorAdmin.tsx` exactly (it uses only `font-bold` and unweighted default).

Nothing else on this page uses the accent gradient or `brandindigo`/`brandfuchsia` — table rows, meta text, and borders stay on the secondary/neutral palette.

## Layout & Component Inventory

Mirrors `components/admin/CuratorAdmin.tsx` (closest structural precedent — invite-style creation form + list of existing records) more closely than `ChecklistAdmin.tsx` (drag-reorder, not applicable here).

### Page (`app/(admin)/admin/members/page.tsx`)
- Server component, same admin-gate boilerplate as `app/(admin)/admin/curators/page.tsx`: `createServerClient()` → `auth.getUser()` → redirect `/signin` if absent → check `user.app_metadata.is_admin === true` → redirect `/` if false. Do not rely on layout-level gating alone (project convention, confirmed in `08-CONTEXT.md` code_context).
- Fetch existing industry members server-side via `createServiceClient()`: `artist_profiles` rows where `member_type = 'industry'`, ordered `created_at DESC`.
- Wrapper: `<div className="flex-1 px-9 py-[30px]">` (verbatim from `/admin/curators`).
- `<h1 className="text-2xl font-bold text-white">Industry Members</h1>`
- Renders `<MembersAdmin initialMembers={...} />` client component.

### Admin layout nav (`app/(admin)/layout.tsx`)
- Add one `<Link>` to the existing sidebar list, same style as the other three: `href="/admin/members"`, label "Industry Members", classes `rounded-lg px-3 py-2 text-[13px] text-white/70 transition hover:bg-white/10 hover:text-white`.

### `MembersAdmin` client component (`components/admin/MembersAdmin.tsx`)
State machine mirrors `CuratorAdmin`: `showAddForm`, `addForm`, `saving`, `addError`, `members` (optimistic prepend on success). No edit/delete state — this route only creates, per D-02/D-04 scope (no update/remove decision exists in CONTEXT.md).

**Collapsed state:**
```
<button className="mb-4 rounded-lg bg-grad px-4 py-2.5 text-[13px] font-bold text-white shadow transition hover:opacity-90">
  Invite industry member
</button>
```

**Expanded form state** (`bg-[#0a0a0f] border border-brandindigo/30 rounded-[10px] p-4`, verbatim container from `CuratorForm`):
- Title: `<h3 className="mb-3 text-[13px] font-bold text-white/70">Invite an industry member</h3>`
- Inline error line (if present): `<p className="mb-3 text-[13px] text-rose-400">{addError}</p>`
- Field 1 — Email (required): standard text input, classes verbatim from `CuratorForm`'s email field (`rounded-lg border border-white/10 bg-white/5 px-3 py-2 text-[14px] text-white placeholder:text-white/30 focus:border-brandindigo/60 focus:outline-none`).
- Field 2 — Display name (required): identical input styling, placeholder "e.g. Jordan Ellis".
- Field 3 — Initial role(s) (required, multi-select): reuse the **exact chip-toggle pattern from `components/profile/ProfileForm.tsx` lines 476–503** — grouped by `INDUSTRY_ROLE_GROUPS` category (Creative / Performance / Technical / Business), each group header `text-[11px] font-bold uppercase tracking-[.18em] text-lavdim`, chips in a `flex flex-wrap gap-2`:
  - Selected: `rounded-full border px-3 py-1.5 text-xs font-semibold transition border-lav/50 bg-lav/20 text-white`
  - Unselected: `rounded-full border px-3 py-1.5 text-xs font-semibold transition border-white/10 bg-white/5 text-white/50 hover:border-white/20 hover:text-white/80`
  - Below the groups, a live count line only when ≥1 selected: `"{n} role{s} selected"`, `text-xs text-white/30`.
  - This directly implements D-04 (invite-time role picks from `INDUSTRY_ROLE_GROUPS`) and feeds both `industry_roles` (TEXT[]) and the mapped `roles` (JSONB) column per D-08's amendment — a UI concern only insofar as the same selected-slug array is submitted once; the dual-write is server-side (`createIndustryMember()`), not a second picker.
- Submit row: `<button disabled={saving}>{saving ? 'Sending invite…' : 'Send invite'}</button>` (bg-white/text-black idle style, verbatim from `CuratorForm`'s save button) + `Cancel` button (verbatim ghost style).

**List of existing industry members** (below the form/CTA):
- Empty state: `<p className="mt-4 text-[14px] text-white/50">No industry members yet. Invite the first one above.</p>`
- Populated state: `flex flex-col gap-2` of cards, each `rounded-xl border border-white/10 p-4` (verbatim from `CuratorAdmin`'s list-item card):
  - Display name: `text-[14px] font-bold text-white`
  - Meta line: `text-[12px] text-lavdim` → `"{email} · Joined {created_at, formatted e.g. 'Jul 4, 2026'}"`
  - Role badges: `flex flex-wrap gap-1 mt-2`, each badge `rounded-full border border-hair bg-card2 px-2 py-0.5 text-[11px] text-lav` (verbatim genre-tag style from `CuratorAdmin`'s `genre_focus` pills) — one pill per selected role label.
  - No action buttons on list rows (no edit/delete/resend decided in CONTEXT.md for this route — do not add one).

### States summary

| State | Visual behavior |
|-------|------------------|
| Empty | CTA button visible; empty-state copy below it; no list rendered |
| Form closed (default, ≥1 member exists) | CTA button + populated list below |
| Form open, idle | Inline form visible per spec above; CTA button hidden |
| Submitting | Submit button shows "Sending invite…", disabled; Cancel remains enabled but has no defined behavior mid-submit (leave default — do not add cancel-during-submit handling, not in scope) |
| Success | Form collapses (`showAddForm` → false), form resets to empty, new member row is optimistically prepended to the list (mirrors `CuratorAdmin`'s `setMembers(prev => [json.data, ...prev])`) — the appearance of the new row IS the success signal; no separate toast/banner component exists in this codebase's admin surface, so none is introduced here |
| Error (validation) | Inline red text above the fields (`text-[13px] text-rose-400`), form stays open, values preserved |
| Error (server, e.g. duplicate email) | Same inline red text treatment, using the server's returned error message (fallback to generic copy above) |
