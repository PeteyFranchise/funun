<objective>
Stand up the relationship-graph schema: the `connections` table (mutual request/accept state machine), the `blocks` table (directional storage, bidirectional check), the `no_block()` SECURITY DEFINER helper that every socially-exposed table's RLS will call, and the `reserved_handles` lookup table seeded against handle squatting.

Purpose: These are the tables Phases 10 (Connect), 11 (Messaging), and 13 (Trust & Safety) build on. Creating `blocks` + `no_block()` now — even though the block *feature* ships in Phase 13 — means later phases inherit RLS block-enforcement for free (D-15). `reserved_handles` blocks impersonation/squatting at handle-claim time (D-13/D-14). This plan creates the tables and the helper only; wiring `no_block()` into existing tables' policies is plan 08-04, and the block button UI is Phase 13.

Output: Migrations `035_connections_blocks.sql`, `037_reserved_handles.sql`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (authenticated JWT) → PostgREST → connections/blocks | Direct PostgREST access must not let a member read others' blocklists or forge a request from another member |
| RLS policy body → no_block() (SECURITY DEFINER) | The function runs as its owner, bypassing blocks RLS to check both directions — its search_path must be un-hijackable |
| handle-claim path → reserved_handles | Squatting/impersonation is prevented only if the lookup is broad and case-insensitive |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-03 | Information Disclosure | A member enumerating who blocked them via direct PostgREST on blocks | high | mitigate | blocks_select_own restricts SELECT to blocker_id = auth.uid(); directional storage + no_block()-only bidirectional check means the blocked party's row is never readable by the blocked party |
| T-08-04 | Elevation of Privilege / Tampering | search-path hijacking of the no_block() SECURITY DEFINER function | high | mitigate | SET search_path = '' with fully-qualified public.blocks; EXECUTE revoked from PUBLIC/anon and granted only to authenticated (RESEARCH SECURITY DEFINER correctness + A2) |
| T-08-05 | Spoofing | Handle squatting / brand impersonation | medium | mitigate | reserved_handles seeded broadly with system + music-platform brand names; claim validation looks up (case-insensitively) against it (D-13/D-14) — wiring the lookup into the claim path is deferred to whatever validates handles, flagged for Phase 9 profile-edit |
| T-08-06 | Tampering | Forged connection request impersonating another requester | medium | mitigate | connections_insert policy WITH CHECK (requester_id = auth.uid()) — a caller can only insert a request as themselves |
</threat_model>
