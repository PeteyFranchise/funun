<objective>
Stand up the capability-grant data model and its two server-side helper modules, plus the Wave 0 unit tests that gate every later plan. This is the schema foundation the entire phase depends on: the `capability_grants` join table (request/approve state machine mirroring `connections`), the `grantCapability()`/`requestCapability()` service functions, and the `hasCapability()` read helper used by both the nav (Plan 03) and D-14 route enforcement (Plan 02).

Purpose: Replace the single exclusive `member_type` value with a set of granted capabilities on one account, modeled as rows so both an instant-grant path and a review-then-grant path can share one table (D-01/D-02). Every existing account is auto-preserved as its one existing grant (D-12) with zero behavior change.
Output: One migration file, two `lib/capabilities/*` modules, two `__tests__/capability-*.test.ts` files, and a schema-push verification step.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated JWT → capability_grants (direct PostgREST) | An authenticated user could try to read others' grants or write an 'approved' row directly, bypassing the API route's business logic |
| service_role (API routes) → capability_grants | Trusted writer; bypasses RLS and column grants — the only path allowed to flip status or attach badges |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-15-01 | Elevation of Privilege | capability_grants direct INSERT with status='approved' | high | mitigate | REVOKE INSERT/UPDATE/DELETE from authenticated/anon in migration 042; all writes go through service_role API routes (Plan 02). hasCapability() filters on status='approved' so a self-inserted 'pending' row grants nothing even if RLS were bypassed |
| T-15-02 | Tampering / Repudiation | Concurrent duplicate capability requests for the same (profile, capability) | medium | mitigate | Partial unique index capability_grants_active_uniq enforces at DB level; grantCapability throws DuplicateCapabilityRequestError on error.code '23505' for the route (Plan 02) to return a friendly 409 |
| T-15-03 | Information Disclosure | Reading another member's grant/request state via direct PostgREST | medium | mitigate | RLS capability_grants_select_own USING (profile_id = auth.uid()); SELECT revoked from anon entirely |
| T-15-04 | Tampering | Backfill migration fails hard against a database that has not yet applied 034–040 (member_type absent) | low | mitigate | information_schema.columns existence guard (Pitfall 3) makes the backfill a safe no-op when member_type is absent |
| T-15-SC | Tampering | npm/pip/cargo installs | low | accept | No new packages introduced this phase (RESEARCH Package Legitimacy Audit: not applicable); nothing to slopcheck |
</threat_model>
