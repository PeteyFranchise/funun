<objective>
Unify the two nav experiences into one capability-aware left sidebar and retire the `(industry)` route group. `ArtistNav` gains a `capabilities` prop and hides rooms the account cannot use (D-08). The layout fetches the capability set server-side and passes it down. The `(industry)` layout's three links (Opportunities, Split Sheets, Post) are redistributed: opportunity pages move into `(artist)` under their current URLs and are surfaced from the Antenna room (D-07); Split Sheets folds into Contract Locker (D-06). A subtle footer entry point (D-09) opens the request flow.

Purpose: Deliver the user-visible half of the capability model — one account, one nav, contextual to what it can do. The nav-hiding here is UI convenience; the real permission boundary lives in Plan 02's server checks (D-08 governs UI only, per D-14).
Output: A capability-aware ArtistNav, a server-wired artist layout, relocated opportunity routes, a retired `(industry)` layout, and the D-09 CTA component.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server layout → client ArtistNav (props) | Capability set crosses from a server component to a client component as a prop; it must be read server-side, never client-fetched |
| client CapabilityCta → /api/capabilities/request | The CTA is UI convenience; the actual grant decision (instant vs pending) is enforced by the Plan 02 route, not this component |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-15-10 | Elevation of Privilege | Sidebar hides a room but its route stays reachable by URL (Pitfall 1) | high | mitigate | Nav-hiding here is UI only (D-08); the enforcing boundary is Plan 02's server-side hasCapability() check (D-14). This plan does not rely on nav visibility for security |
| T-15-11 | Information Disclosure | Client-side fetch of capability_grants leaking another account's request state | medium | mitigate | The capability set is read exclusively in the server-component layout via createServiceClient and passed as a prop; no client fetch against capability_grants exists |
| T-15-12 | Spoofing | CapabilityCta POSTing a target profile_id to grant another account | high | mitigate | The CTA sends only `{ capability, role_slugs }`; the request route derives the target from the session (Plan 02 T-15-05), so a forged profile_id is impossible |
| T-15-SC | Tampering | npm/pip/cargo installs | low | accept | No new packages introduced (RESEARCH: no new library surface); nothing to slopcheck |
</threat_model>
