<objective>
Build the two capability API routes and close the D-14 server-side enforcement gap. The request route implements D-02's asymmetric gate (industry→artist instant, artist→industry pending). The approve route extends the `/api/admin/members` pattern with a `verifyAdmin()` gate to flip pending requests to approved and attach the pre-picked badge. The opportunities POST route gains a real `hasCapability()` check so nav-hiding (Plan 03) is backed by an actual permission boundary, not just UI.

Purpose: Make "capability" a real, requestable, combinable server-enforced permission — not the pre-existing nav-only boundary. This is the layer that turns the Plan 01 data model into a working request/approve/enforce flow.
Output: Two new route handlers, one modified route (D-14 enforcement), and a route-guard test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /api/capabilities/request | Untrusted body; a client could try to supply a target profile_id or a forged 'approved' status |
| client → /api/capabilities/approve/[grantId] | Only admins may cross; a non-admin could try to self-approve their own pending industry request |
| client → /api/antenna/opportunities (POST) | Any authenticated account can reach it today with zero capability check (pre-existing gap, D-14 target) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-15-05 | Elevation of Privilege | /api/capabilities/request accepting a client profile_id to grant another account | high | mitigate | Requester target is always `auth.getUser().id`; the route never reads profile_id/profileId from the body (V4). requestCapability signs the grant to the session user only |
| T-15-06 | Elevation of Privilege | Non-admin flipping their own pending industry request to approved | high | mitigate | approve route's first statement is verifyAdmin() (403 for non-admins); the target profile_id comes from the loaded grant row, not the caller |
| T-15-07 | Elevation of Privilege | Artist-only account posting an opportunity via direct POST despite a hidden nav (Pitfall 1) | high | mitigate | D-14: hasCapability(user.id,'industry') server-side check returns 403; nav-hiding (Plan 03) is defense-in-depth, not the boundary |
| T-15-08 | Input Validation | Malformed capability/role_slugs/decision body | medium | mitigate | isValidCapability strict literal guard, isValidRoleSlugList allowlist, decision ∈ {'approve','deny'} literal check — all reject → 400 |
| T-15-09 | Tampering / Repudiation | Double-decide a grant (approve an already-approved request) | low | mitigate | approve route returns 409 when `status !== 'pending'`; decided_by/decided_at record who/when for audit |
</threat_model>
