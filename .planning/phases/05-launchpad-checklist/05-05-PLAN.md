Build the six admin API endpoints behind the `is_admin` gate: full CRUD + atomic reorder for checklist items, and the tip draft list + approve/reject workflow. Each route independently verifies `is_admin` (the layout redirect is not sufficient), then uses the service-role client for cross-user writes. These satisfy LAUNCH-05 (admin item CRUD/reorder/delete) and the admin half of LAUNCH-03 (approve tips before they publish).

Purpose: The admin UI (Plan 06) is a thin client over these endpoints. All authority decisions (admin gate, field allowlist, tip promotion) live here.
Output: 4 route files covering 6 HTTP handlers.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| any user → /api/admin/* | Only is_admin users may list, create, edit, delete, or approve |
| admin client → checklist PATCH | Only allowlisted fields may be written (no mass assignment) |
| request param → SQL | itemKey flows into a WHERE clause and must be constrained |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-02 | Elevation of Privilege | every /api/admin/* handler | critical | mitigate | verifyAdmin() runs first on every handler (401 no-user / 403 non-admin); createServiceClient() is only invoked after the gate passes — the layout redirect alone is treated as insufficient per RESEARCH anti-patterns |
| T-05-07 | Tampering | checklist item PATCH body | high | mitigate | EDITABLE_FIELDS allowlist; only label, section, action_type, action_href, action_label, sort_order are writable; section/action_type validated against fixed literals |
| T-05-08 | Tampering | itemKey in WHERE/DELETE path | medium | mitigate | itemKey validated against `/^[a-z0-9_]+$/` before use; Supabase client parameterizes queries |
| T-05-05 | Information Disclosure | tip approval | high | mitigate | tip_draft only becomes visible to artists (Plan 03 GET) after approve sets tip_approved=true; reject clears the draft without publishing |
| T-05-SC | Tampering | npm installs | low | accept | No package installs in this plan |
</threat_model>
