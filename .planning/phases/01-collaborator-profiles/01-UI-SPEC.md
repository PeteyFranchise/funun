| Body / input text | 14px (`text-sm`) | 400 (regular) | 1.5 | Form inputs, card body copy, picker search input — standard prose size |

**Weight justification (4 weights):**

| Weight | Tailwind | Used For |
|--------|----------|----------|
| 400 (regular) | `font-normal` | Body text, input values, card sub-text |

Label casing: ALL CAPS with `tracking-wide` (`tracking-[.12em]` to `tracking-[.18em]`) — matches every existing form label in the codebase.

## Component Contracts

### CollaboratorCard

Layout: fixed-height card, border-radius `rounded-card` (18px), `border-hair`, `bg-card`.

Anatomy (top to bottom):
1. Name — `text-[14.5px] font-bold text-white`
2. PRO affiliation — `text-[12.5px] text-lavdim` (show PRO label from `PRO_LABELS`, or "No PRO" if `pro === 'none'`)
3. IPI status badge — one of the semantic badges from the table above; inline-flex, rounded-full, `px-3 py-1`
4. Edit trigger — ghost button, bottom-right, `text-xs text-white/50 hover:text-white`

States:
- Normal (IPI present): shows "IPI on file" badge
- Warning (IPI missing): shows "IPI missing" amber badge; card left-border accent `border-l-2 border-l-amber-400/70` matching `DocumentCard` warning pattern
- No PRO: PRO line reads "No PRO on file" in `text-lavdim`

Touch target: edit button minimum 32px height.

### CollaboratorForm (modal overlay)

Pattern: identical to `EditProjectForm` — `useState(open)` toggle, form renders inline (replaces button), `rounded-xl border border-white/10 bg-white/[0.03] p-4`.

Fields (in order, 2-column grid on sm+):
1. Full legal name * (required, sm:col-span-2)
2. Email (sm:col-span-1)
3. Phone (sm:col-span-1)
4. PRO affiliation — `<select>` using `PRO_VALUES` / `PRO_LABELS` (sm:col-span-1)
5. IPI/CAE number (sm:col-span-1)
6. Publisher (sm:col-span-1)
7. MLC member ID (sm:col-span-1)
8. SoundExchange ID (sm:col-span-1)
9. Mailing address — single textarea, `resize-none`, rows=2 (sm:col-span-2); label: "Mailing address (optional)"

All inputs use the established class: `w-full rounded-lg border border-white/10 bg-white/5 px-3 py-2 text-sm text-white placeholder-white/30 outline-none focus:border-white/30`.

All labels use: `block text-xs font-medium uppercase tracking-wide text-white/40`.

Buttons (bottom row, `flex items-center justify-between`):
- Primary: "Save collaborator" — `rounded-lg bg-white px-3 py-1.5 text-sm font-semibold text-black hover:bg-white/90 disabled:opacity-40`
- Cancel: `text-sm text-white/50 hover:text-white`
- Delete (existing collaborators only): initially "Delete collaborator" ghost-danger; on first click shows inline confirmation "Delete [Name]?" with "Yes, delete" (`bg-rose-500/90`) and "Keep" cancel — matches `EditProjectForm` confirm-delete pattern exactly

### CollaboratorPicker

A compact picker component embedded inside `ComposerEditor` and `SplitSheetBuilder` signer rows.

Trigger: small button rendered to the right of the name input in a composer row:
- Label: "Pick from roster"
- Style: `rounded-lg border border-dashed border-white/15 px-2 py-1 text-xs text-white/50 hover:border-white/30 hover:text-white` (matches the "+ Add writer" dashed button style)
- When roster is empty: button reads "Add collaborator" and opens form directly

Dropdown panel (absolute positioned below trigger):
- Surface: `bg-card border border-hairstrong rounded-xl shadow-xl`
- Width: min 240px, max 320px
- List items: `px-4 py-2 text-sm text-white hover:bg-white/5 cursor-pointer`; show name + PRO label in smaller `text-lavdim` below
- Search input at top of panel: same input class, `placeholder="Search collaborators…"`
- Bottom of list: divider `border-t border-hair` then "Add new collaborator" item in `text-brandindigo text-sm font-medium`
- Keyboard: arrow keys navigate list, Enter selects, Escape closes

Inline "Add new collaborator" flow: clicking the bottom item opens the `CollaboratorForm` rendered inline within the picker panel (not a new modal). On save, the new collaborator is selected automatically.

IPI-missing flag on row (D-03):
- After picker auto-fills a row where IPI is absent, append a warning chip to that composer row: amber badge "IPI missing — complete before export"
- Style: `inline-flex items-center gap-1 rounded-full border border-amber-400/30 bg-amber-400/10 px-2 py-0.5 text-[11px] font-bold text-amber-300`
- Chip sits below the row inputs, left-aligned

"Save to profile?" nudge (D-07):
- Appears when user edits IPI, PRO, publisher, email, or phone at the song level after picking from roster
- Treatment: small inline icon button at the right edge of the modified field — a cloud-upload icon (SVG, 14×14, `text-white/30 hover:text-white/70`) with tooltip on hover: "Save to [Name]'s profile?"
- On click: fires PATCH to collaborator profile; shows brief "Saved" text in `text-emerald-300 text-xs` for 2s then disappears
- Split field: never shows this nudge (D-07 — split is always song-specific)

### SplitSheetBuilder

Layout: full-page form (not modal), accessible via standalone `/split-sheets/new` route (artist) or industry equivalent.

Sections:
1. Song name field (required) — full-width input
2. Linked project (optional) — `<select>` of artist's vault projects, or "Not tied to a release" option
3. Parties section — "Writers & Split" heading with "+ Add party" button (dashed, same as "+ Add writer")
   - Each party row: mirrors `ComposerEditor` grid — `CollaboratorPicker` trigger | Name input | Role select | PRO select | IPI | Split % | Remove button
   - Role select uses `COMPOSER_ROLE_VALUES` / `COMPOSER_ROLE_LABELS`
   - Split % shows live running total: "Total: X%" — green (`text-emerald-300`) when 100, amber otherwise

4. Action row: "Save draft" (ghost) | "Send for approval" (primary gradient CTA)

"Send for approval" gate: button is disabled and shows tooltip "All splits must total 100% before sending" if total ≠ 100.

### SplitApprovalView (public page)

Layout: centered, max-width 480px, on top of `bg-ink`.

Header: Funūn wordmark (text-only, `gtext` gradient) + "Split approval request from [Artist Name]"

Content card (`bg-card border-hair rounded-card p-6`):
- Song name — `text-[18px] font-extrabold text-white`
- All parties listed with their proposed split %:
  - Each row: name + role + "X%" — current collaborator's row highlighted with `bg-card2`
- Total line: "Total: 100%" in `text-emerald-300 font-bold` (always 100 at send-time)

Action section (below card):
- Primary: "Approve this split" button — `bg-grad` style
- Secondary: "Propose a different split" toggle — reveals inline counter-proposal input

  - Validation note (inline, below input): "The remaining X% will be distributed among other parties. The initiating artist will be notified to review and re-send."
  - Note: the system only captures this collaborator's counter-proposal %; it does NOT auto-redistribute others (artist iterates manually per D-16)
  - Submit button for counter: "Submit counter-proposal"

Expired/used token state:
- Card reads: "This approval link has expired or has already been used."
- Sub-copy: "Ask [Artist Name] to send a new approval link."
- No action buttons

## Interaction Contracts

### Picker Auto-fill Sequence

1. Artist clicks "Pick from roster" on a composer row.
2. Picker dropdown opens with search input focused.
3. Artist types or scrolls to select a collaborator.
4. On selection: name, PRO, IPI, email, phone fields auto-fill. Role and split remain unchanged.
5. If IPI is missing: amber "IPI missing" chip appears below the row immediately.
6. Dropdown closes.
7. If IPI was missing: on track save, API sends missing-IPI emails to artist + collaborator (not on picker action — on save only, per D-03).

### Edit Collaborator Sequence

1. Artist clicks "Edit" on a collaborator card.
2. `CollaboratorForm` modal opens inline (same page, following `EditProjectForm` pattern).
3. Artist edits fields and clicks "Save collaborator".
4. PATCH `/api/collaborators/[id]` fires.
5. `router.refresh()` revalidates the roster.
6. Form closes, updated card appears.

### Split Approval Iteration Sequence

1. Artist finalizes parties and splits in `SplitSheetBuilder`.
2. Clicks "Send for approval" — server generates tokens per party, sends emails.
3. Each collaborator receives email with approval link to `/approve/[token]`.
4. Collaborator opens link (no auth required), sees proposed split, approves or counter-proposes.
5. On approve: collaborator's row status → "Approved"; artist notified by email.
6. On counter: collaborator's row status → "Countered"; artist sees counter % and can revise the split sheet and re-send.
7. All rows "Approved" → split sheet status → "Approved"; `all_approved_at` timestamp set.

### IPI Invite Sequence

1. Artist saves a track with a composer row populated from a collaborator whose IPI is missing.
2. Track save API handler checks: was the missing-IPI email already sent for this collaborator+artist pair within 24 hours? If yes, skip. If no, proceed.
3. API creates a `collaborator_invites` record with a 30-day token.
4. Two emails sent via `sendEmail()`:
   - To collaborator: educational IPI email + invite link to `/signup?invite=[token]`
   - To artist: "Here's what to do once your collaborator has their IPI"
5. After collaborator signs up via invite link, middleware detects the `invite` param, connects the new account to the invite record, redirects to `/join/[inviteToken]` view-only profile.

## Accessibility Contract

- All form inputs have an associated `<label>` (htmlFor) — not just placeholder text.
- Badge-only states (IPI missing card) must also have a text label, not color alone. The amber badge text "IPI missing" satisfies this; do not rely on color alone for IPI status.
- Picker dropdown uses `role="listbox"` and `role="option"` on list items; trigger has `aria-expanded` and `aria-haspopup="listbox"`.
- Approval page "Approve" and "Submit counter-proposal" buttons must have distinct, descriptive labels (no "Submit" generic labels).
- Minimum touch target for all interactive controls: 32px height (existing project standard).
- Public pages (`/approve/[token]`, `/join/[inviteToken]`) must render legibly without JavaScript (server-rendered; forms degrade gracefully).
