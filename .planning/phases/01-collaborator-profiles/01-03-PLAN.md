Then insert split_sheet_parties rows (denormalized snapshot of name/email/pro/ipi/role/split_percentage + collaborator_id) for the new split_sheet_id. Return { data: { id, ... } }. On any DB error return 500 with the message.

Return { data }.
    - DELETE(_request, {params}): auth check; delete split_sheets .eq('id', params.id).eq('initiator_user_id', user.id) (parties cascade via FK). Return { ok: true }.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → POST /api/split-sheets | Untrusted song name + party array crosses into the split-sheet layer |
| user → another user's split sheet | Edit/delete attempts on a sheet the user did not initiate |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-07 | Tampering | POST/PATCH /api/split-sheets | high | mitigate | Party-field allowlist + server-side `validateApprovalTotal` (reject totals ≠ 100); initiator_user_id sourced from session, never body |
| T-01-08 | Elevation of Privilege | PATCH/DELETE /api/split-sheets/[id] | high | mitigate | Handlers scope to `.eq('initiator_user_id', user.id)`; RLS "Initiator manages split sheet" is the backstop |
| T-01-09 | Information Disclosure | split_sheets / split_sheet_parties reads | medium | mitigate | RLS policies restrict reads to initiator or named party (auth.uid()); GET filters by initiator_user_id |
| T-01-SC | Tampering | npm installs | low | accept | No new packages introduced this phase (RESEARCH.md: zero new dependencies); token generation uses Node built-in crypto |
</threat_model>
